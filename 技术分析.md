# CompIntel 多 Agent 协同通信架构技术分析

> 本文档分析 CompIntel Research 项目中 8 个 Agent 如何通过 LangGraph StateGraph 实现协同通信、并行执行、聚合收敛和闭环反馈。

---

## 1. 总体拓扑

### 1.1 通信载体：共享状态字典

所有 Agent 之间不直接调用、不发送消息、不通过消息队列。唯一的通信方式是一个 **共享的 `TypedDict` — `CompIntelState`**（定义在 `state.py:36`）。

```
Query
  │
  ▼
┌─────────────────────────────────────────────────────────┐
│              CompIntelState (共享 TypedDict)             │
│                                                         │
│  写入者: 每个 Agent 返回增量 dict，LangGraph 自动 merge   │
│  读取者: 每个 Agent 通过 StateAdapter 读取所需字段        │
│  聚合者: Annotated[list, operator.add] 声明的字段自动拼接  │
└─────────────────────────────────────────────────────────┘
  │
  ▼
[intent_analyst]  ──► [research_planner]  ──► [fan-out: N× competitor_profiler]
  │                                                    │
  │                                          (内部子图: 3 workers 并行)
  │                                                    │
  ▼                                                    ▼
[market_analyst]  ←──────────── profiles 自动聚合 ──────┘
  │
  ▼
[swot_synthesizer]  ──► [report_writer]  ──► [reviewer]
                                                    │
                                    ┌───────────────┴──────────────┐
                                    │ approved (score >= 7)        │ revise (score < 7)
                                    ▼                              ▼
                              [rag_ingest]                   [report_writer]
                                    │                         (retry ≤ 3 次)
                                    ▼
                                   END
```

### 1.2 两套状态定义

| 状态类 | 作用域 | 定义位置 |
|--------|--------|---------|
| `CompIntelState` | 主 pipeline（8 个 Agent 节点） | `state.py:36` |
| `CompetitorProfileGraphState` | 每个竞品内部的 3-worker 子图 | `graph.py:27` |

两套状态互不干扰。子图运行结束后，`profile_app.ainvoke()` 返回结果字典，由主图的 `_profile_one_node` 取出需要的字段写入 `CompIntelState`。

## 2. 通信协议三层架构

### Layer 1: LangGraph 图引擎（路由层）

`CompIntelGraph`（`graph.py:45`）是全局编排器，持有所有 Agent 实例和两套 StateGraph。

```python
# graph.py:193-217 — 主编排图
graph = StateGraph(CompIntelState)
graph.add_node("intent_analyst", self._intent_node)
graph.add_node("research_planner", self._planner_node)
graph.add_node("competitor_profiler", self._profile_one_node)  # 扇出目标
# ...
graph.add_edge(START, "intent_analyst")
graph.add_edge("intent_analyst", "research_planner")
graph.add_conditional_edges("research_planner", self._fan_out_competitors)  # ★ 条件扇出
graph.add_edge("competitor_profiler", "market_analyst")  # ★ 扇出后自动汇聚
# ...
graph.add_conditional_edges("reviewer", self._review_route,  # ★ 条件回环
    {"approved": "rag_ingest", "revise": "report_writer"})
```

**路由能力**：
- `add_edge(A, B)` — 固定顺序流转
- `add_conditional_edges(A, fn, mapping)` — 基于状态值的动态路由

### Layer 2: StateAdapter（访问层）

每个 Agent 不再直接访问 `state.get("xxx")` 散布，而是通过 `StateAdapter` 统一读写。

```python
# agents/base.py:57 — BaseCompIntelAgent.read_state()
def read_state(self, state: Any) -> StateAdapter:
    return StateAdapter(state if isinstance(state, dict) else {})
```

StateAdapter 提供 30+ 类型化属性，每个属性内置空值防护：

```python
# core/state_adapter.py
s.query          → str,   默认 ""
s.profiles       → list,  默认 []
s.market_segment → str,   自动回退 intent.market_segment
s.intent         → dict,  默认 {}
s.retry_count    → int,   默认 0
```

### Layer 3: LLMService（能力层）

每个 Agent 通过 `self.llm` 属性懒加载获取 `LLMService` 实例，不再各自维护 `completion_fn`、`_split_provider_model`、重试逻辑。

```python
# agents/base.py:48
@property
def llm(self) -> LLMService:
    if self._llm is None:
        self._llm = LLMService(CompIntelSettings.from_env())
    return self._llm
```

LLMService 统一提供三个调用接口：

| 方法 | 用途 | 内部行为 |
|------|------|---------|
| `call()` | 纯文本生成 | HTTP 层 3 次重试 (0.5→1→2s 退避) |
| `call_and_parse()` | JSON 结构化输出 | call() + json_repair 解析 + 解析失败重试 |
| `call_with_degradation()` | 三级降级 | LLM → derived_fn → template_fn |

## 3. Fan-out / Fan-in 并行机制

### 3.1 主图级扇出：Send 机制

当 `research_planner` 识别出 N 个竞品后，`_fan_out_competitors` 返回 `list[Send]`，LangGraph 为每个 Send 创建一个独立的并行执行分支：

```python
# graph.py:240-254
def _fan_out_competitors(self, state) -> list[Send] | Literal["market_analyst"]:
    competitors = state.get("competitors", [])
    if not competitors:
        return "market_analyst"  # 无竞品 → 跳过
    return [
        Send("competitor_profiler", {
            "competitor": competitor,
            "research_questions": state.get("research_questions", []),
            "market_segment": state.get("market_segment", ""),
        })
        for competitor in competitors
    ]
```

**关键设计**：每个 Send 传递的是**独立的状态增量**，不包含全局的路由信息。LangGraph 保证：
- N 个 Send 并行执行，互不阻塞
- 每个 Send 有自己的一份 `{competitor, research_questions, market_segment}` 副本
- 执行完成后自动汇聚到 "competitor_profiler" → "market_analyst" 边

### 3.2 聚合：Annotated Reducer

所有并行分支的产出通过 `Annotated[list, operator.add]` 自动拼接：

```python
# state.py:46
profiles: Annotated[list[dict[str, Any]], operator.add]

# state.py:55
execution_log: Annotated[list[ExecutionEvent], operator.add]
```

`operator.add` 在 list 上等同于 `list1 + list2`（拼接）。当 N 个 `competitor_profiler` 各自返回 `{"profiles": [profile_i]}`，LangGraph 自动将所有 `[profile_1] + [profile_2] + ... + [profile_N]` 合并为 `[profile_1, profile_2, ..., profile_N]`。

### 3.3 容错隔离

```python
# graph.py:256-284
async def _profile_one_node(self, state) -> dict:
    try:
        result = await self.profile_app.ainvoke(...)
        profile = result.get("profile", {})
        return {"profiles": [profile], ...}
    except Exception as exc:
        name = state.get("competitor", {}).get("name", "unknown")
        return {
            "profiles": [{"name": name, "summary": f"{name} profiling skipped"}],
            "warnings": [f"{name} profiling skipped — source unavailable..."],
        }
```

**一个竞品失败不阻塞其余竞品**。因为 LangGraph 的 Send 将每个分支视为独立执行单元，exception 被 try/except 捕获后返回占位 profile，Annotated reducer 仍然将其与成功的 profiles 合并。

## 4. 子图级并行：内部 3-Worker Fan-out

### 4.1 子图结构

每个 `competitor_profiler` 实例内部有一个独立的 `profile_app` StateGraph：

```python
# graph.py:176-191 — 子图
def _build_profile_graph(self):
    graph = StateGraph(CompetitorProfileGraphState)
    graph.add_node("fan_out", lambda s: {})       # 空节点，仅做分发源
    graph.add_node("search_worker", ...)
    graph.add_node("scrape_worker", ...)
    graph.add_node("rag_retriever", ...)
    graph.add_node("aggregator", ...)             # 汇聚节点

    graph.add_edge(START, "fan_out")
    graph.add_edge("fan_out", "search_worker")    # ─┬─ 三条并行边
    graph.add_edge("fan_out", "scrape_worker")    #  ├─ 同时出发
    graph.add_edge("fan_out", "rag_retriever")    # ─┘
    graph.add_edge("search_worker", "aggregator") # ─┬─ 全部汇聚到
    graph.add_edge("scrape_worker", "aggregator")  #  ├─ aggregator
    graph.add_edge("rag_retriever", "aggregator")  # ─┘
    graph.add_edge("aggregator", END)
    return graph.compile()
```

**设计**：一个空 `fan_out` 节点连接 3 条边到 3 个 worker。LangGraph 遇到多条出边时并行执行它们。3 个 worker 完成后各自连接到 `aggregator`，形成汇聚点。

### 4.2 子图 Worker 职责

| Worker | 输入 | 输出 | 数据来源 |
|--------|------|------|---------|
| `search_worker` | competitor + research_questions | `search_results` | Tavily / SerpAPI 搜索 |
| `scrape_worker` | competitor (name + website) | `scraped_content` | 官网 + G2/Capterra |
| `rag_retriever` | competitor + market_segment | `rag_context` | Qdrant 向量库 |

每个 worker 在 `competitor_profiler.__call__` 中被独立 try/except 包裹，单个 worker 失败不会影响其他 worker：

```python
# competitor_profiler.py:32-46
try:
    search = await self.search_worker(...)
except Exception as exc:
    logger.warning("Search worker failed for %s: %s", name, exc)
    search = {}  # 空结果，不阻塞
```

### 4.3 子图 Aggregator

```python
# graph.py:319-340
async def _profile_aggregator_node(self, state):
    competitor = state.get("competitor") or {}
    search = state.get("search_results") or {}
    scrape = state.get("scraped_content") or {}
    rag = state.get("rag_context") or {}

    profile = {
        "name": name,
        "summary": self._build_profile_summary(name, search, scrape, rag),
        "search_results": search.get("search_results", []),
        "scraped_content": scrape.get("scraped_content", []),
        "rag_context": rag.get("rag_context", []),
        "sources": self._extract_profile_sources(search, scrape, rag),
    }
    return {"profile": profile, ...}
```

Aggregator 是三路数据的汇聚点——拼装 search、scrape、rag 的结果为统一的 `profile` 字典。

## 5. 闭环反馈：Reviewer → ReportWriter 回环

### 5.1 条件路由

```python
# graph.py:477-481
def _review_route(self, state) -> Literal["approved", "revise"]:
    feedback = state.get("review_feedback", {})
    if feedback.get("approved") or int(state.get("retry_count", 0)) >= 3:
        return "approved"
    return "revise"
```

这是整个 pipeline 中唯一的**回环**——当 reviewer 打分 < 7 时，路由回到 `report_writer`，让其根据 `review_feedback` 修改报告。

### 5.2 状态驱动的重写

报告 writer 接收 `review_feedback` 作为输入，将其注入 LLM prompt：

```python
# report_writer.py:231 (executive summary prompt 中包含)
f"Prior review feedback: {safe_json_dumps(review_feedback)[:200]}\n"
```

这样 reviewer 的批评直接指导 writer 的重写方向。

### 5.3 防死循环

```python
# graph.py:460-464
retry_count = int(state.get("retry_count", 0))
if not feedback.get("approved"):
    retry_count += 1

# P1-3: 3 次仍未通过 → 自动通过 + banner
if retry_count >= ReviewerAgent.MAX_RETRIES and not feedback.get("approved"):
    feedback["review_banner"] = "本报告在 3 次修订尝试后自动通过..."
```

`retry_count` 单调递增，`MAX_RETRIES = 3`。到第 3 次后 `_review_route` 强制走 `approved` 分支，进入 `rag_ingest`。

## 6. 闭环反馈：RAG 自我记忆循环

### 6.1 写入路径

每次经 reviewer 批准的报告通过 `_rag_ingest_node` 写入 Qdrant 向量库：

```python
# graph.py:483-565
async def _rag_ingest_node(self, state):
    # 1. 提取报告的 executive summary
    documents.append({"text": "Executive Summary: ...", "metadata": {...}})

    # 2. 提取每位竞品的 SWOT
    for comp in swot.get("competitors", []):
        documents.append({"text": f"SWOT for {comp}: ...", "metadata": {...}})

    # 3. 提取市场分析
    documents.append({"text": f"Market analysis: ...", "metadata": {...}})

    # 4. 写入 Qdrant
    self.competitor_profiler.rag_retriever.store.ingest(documents)
```

### 6.2 读取路径

下次查询时，`RAGRetriever` 从 Qdrant 中查询历史分析：

```python
# rag_retriever 伪代码
query_vector = embedder.embed(competitor_name + " " + market_segment)
results = qdrant_store.similarity_search(query_vector, top_k=5)
return results  # 包含上次分析的 SWOT、市场趋势、竞品叙事
```

### 6.3 质量分级

```python
# graph.py:511
quality = "limited" if is_auto_approved else "reviewed"
```

- `"reviewed"` — reviewer 评分 ≥ 7 的报告，高质量记忆
- `"limited"` — auto-approved (3 次重试仍未达标)，可被未来查询降权

## 7. Agent 通信的六个核心模式

### 模式 1：写入-读取（Write-Read）

每个 Agent 读取上游写入的 state 字段，产出新字段写入 state。

```
intent_analyst 写入 "competitors"
    ↓
research_planner 读取 "competitors"，写入 "research_plan"
    ↓
market_analyst 读取 "profiles"，写入 "market_analysis"
```

### 模式 2：扇出-聚合（Fan-out / Fan-in）

所有并行执行的分支写入同一个 Annotated reducer 字段，自动聚合。

```
competitor_profiler[N] 各自写入 {"profiles": [profile_i]}
    ↓ (operator.add 自动拼接)
market_analyst 读取 profiles = [profile_1, profile_2, ..., profile_N]
```

### 模式 3：回环-重试（Retry Loop）

状态驱动条件路由，同一节点可被多次执行直到状态满足退出条件。

```
reviewer → (score < 7 且 retry < 3) → report_writer → reviewer → ...
reviewer → (score >= 7 或 retry >= 3) → rag_ingest → END
```

### 模式 4：注入-反馈（Injection）

上游节点的输出作为下游节点的 prompt 注入，指导 LLM 行为。

```
reviewer 写入 "review_feedback"
    ↓
report_writer 将 review_feedback 注入 LLM prompt
```

### 模式 5：检查点-持久（Checkpoint）

LangGraph 的 checkpointer（MemorySaver 或 SqliteSaver）在每次状态变更后自动持久化快照。

```python
# graph.py:217
return graph.compile(checkpointer=self.checkpointer)

# graph.py:567-569
def _config(self, query):
    thread_id = f"compintel:{hashlib.sha256(query.encode()).hexdigest()[:16]}"
    return {"configurable": {"thread_id": thread_id}}
```

- `thread_id` 用 SHA-256 哈希生成，避免碰撞
- 不同查询 = 不同 thread，checkpointer 隔离
- `SqliteSaver` 持久化到磁盘（`compintel_checkpoints.db`），支持进程重启后恢复

### 模式 6：自记忆循环（Self-Memory Loop）

完成的分析被写入向量库，下次查询自动检索，形成"经验积累"。

```
本次: query → ... → report → rag_ingest → Qdrant
下次: query → rag_retriever → Qdrant → 上次分析作为上下文注入
```

## 8. 与经典 Agent 通信模式对比

| 维度 | CompIntel (LangGraph) | 经典消息传递 (AutoGen/CrewAI) | Pub/Sub 事件总线 |
|------|-----------------------|-----|-----|
| 通信载体 | 共享 TypedDict + reducer | Agent-to-Agent 消息 | Topic/channel |
| 路由方式 | 图边 + 条件函数 | Agent 内部决策/编排器 | 订阅匹配 |
| 并发模型 | Send fan-out (LangGraph 管理) | 手动并发 Dispatch | 事件驱动 |
| 状态持久 | 内置 Checkpointer | 需自行实现 | 需自行实现 |
| 容错 | try/except + Annotation reducer | 需自行实现 | 需自行实现 |
| 调试可观测性 | state snapshots + execution_log | 消息历史 | 事件日志 |
| 适合场景 | 固定流程 + 条件分支 | 自由对话/协商 | 松耦合微服务 |

## 9. 实际运行流举例

以用户查询 `"分析 Notion 在协作软件市场的竞品"` 为例：

```
1. START → app.ainvoke({"query": "分析 Notion...", ...}, config)

2. [intent_analyst] 读取 state.query
   写入 state = {
     "target": "Notion",
     "market_segment": "collaboration software",
     "competitors": [{"name":"Coda"}, {"name":"Confluence"}, {"name":"ClickUp"}],
     "research_questions": ["Notion 产品功能对比", ...],
   }

3. [research_planner] 读取 state.competitors
   写入 state = {
     "research_plan": {"Notion": {...}, "Coda": {...}, ...}
   }

4. [_fan_out_competitors] 读取 state.competitors (3 个)
   返回 [Send("competitor_profiler", {"competitor": "Coda", ...}),
          Send("competitor_profiler", {"competitor": "Confluence", ...}),
          Send("competitor_profiler", {"competitor": "ClickUp", ...})]
   → LangGraph 并行执行 3 个 competitor_profiler 实例

5. [competitor_profiler × 3] 每个内部运行 3-worker 子图:
     Coda 实例:   search → "Coda pricing 2026..."
                  scrape → coda.io 官网内容
                  rag    → 上次的 "Coda vs Notion" 分析
                  聚合 → profile_Coda

     Confluence 实例: search + scrape + rag → profile_Confluence
     ClickUp 实例:    search + scrape + rag → profile_ClickUp

   Annotated[list, operator.add] 自动合并:
   state.profiles = [profile_Coda, profile_Confluence, profile_ClickUp]

6. [market_analyst] 读取 state.profiles (3 个完整 profile)
   LLM 调用: "Analyze the market segment 'collaboration software'..."
   写入 state = {"market_analysis": {overview, trends, landscape, ...}}

7. [swot_synthesizer] 读取 profiles + market_analysis
   LLM 调用: 为每个竞品生成 SWOT + 交叉分析
   写入 state = {"swot_analysis": {competitors: [{strengths, weaknesses, ...}], ...}}

8. [report_writer] 读取 profiles + market_analysis + swot_analysis + review_feedback
   4 次小 LLM 调用:
     - executive_summary ({target} + market overview)
     - competitor_narrative × 3 (并行, Semaphore(3))
     - conclusion (narrative summaries + SWOT)
   纯代码装配 → 写入 state = {"report": {title, sections, conclusion, ...}}

9. [reviewer] 读取 state.report
   LLM-as-Judge 评分: completeness×0.4 + accuracy×0.4 + actionability×0.2
   写入 state = {"review_feedback": {score: 8.2, approved: True, ...}}

10. [_review_route] 读取 review_feedback.approved == True
    路由 → "approved"

11. [rag_ingest] 将报告拆分为 8 个文档，写入 Qdrant
    写入 execution_log: "Ingested 8 chunks into RAG memory for Notion"

12. END → app.ainvoke() 返回完整 state
```

总耗时（实际测量，3 竞品）：~6 分 30 秒（含 LLM 调用和网络 IO）

## 10. 关键设计决策总结

| 决策 | 理由 |
|------|------|
| **共享状态而非消息传递** | 固定 pipeline 流程不需要 agent 自主协商；共享状态更可观测、可调试、可恢复 |
| **Annotated Reducer 聚合** | 避免竞态条件；operator.add 是幂等拼接 |
| **两层扇出** | 主图级 Send（竞品间）+ 子图级并行边（worker 间），两层隔离，互不阻塞 |
| **回环有限次** | reviewer→writer 最多 3 次，防止死循环；auto-approve 保证 pipeline 一定终止 |
| **StateAdapter 统一访问** | 消除 `isinstance(state, dict)` 散布和 `.get()` 链，类型安全访问 30+ 字段 |
| **LLMService 统一调 LLM** | 消除 5 个 agent 的粘贴复制；重试、解析、降级在一处管理 |
| **Checkpointer 隔离** | SHA-256 thread_id，不同查询不污染 |
| **RAG 自记忆** | 每次批准的报告自动入库，下次检索；quality 标记区分人工/自动通过 |
