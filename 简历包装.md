- **RAG 自增长记忆**：Qdrant + BGE-small-zh 本地 Embedding（512 维，CPU 推理，零 Docker），审核通过后自动将报告/SWOT/市场分析写回向量库——从只读知识库升级为读写记忆系统，二次查询优先命中历史洞察而非种子数据。Embedding 可插拔（协议抽象 + 语义/轻量双实现），Top-5 检索相关率 0/5 → 5/5

- **Agent 评估**：自建 7 维度自动评估框架（竞品真实性 / 赛道准确性 / RAG 记忆利用 / Source 可追溯 / 降级行为 / Pipeline 完整性 / 报告可读性），纯规则评分。全链路按节点追踪，报告不达标时从 Reviewer 分项评分反查上游逐环节定位。42 测试覆盖全链路 + 降级双路径

## 面试追问预测（基于第三版简历，逐条推演）

### 模块一：LangGraph 多 Agent 编排

**Q1：「为什么选 LangGraph 而不是 CrewAI 或 AutoGen？」**

这是 AI Agent 开发岗最高频问题，预期每个候选人都会被问到。

**回答要点**：
- LangGraph 给我显式的 State 管理 + `send()` 并行 + Checkpointer 持久化，比 CrewAI 的 Task 抽象更底层可控
- CrewAI 适合快速原型，但竞品分析场景需要精确控制每个 Agent 的输入输出 Schema、需要并行子图、需要断点续跑——LangGraph 的 StateGraph 比 CrewAI 的 Task/Crew 抽象更适合
- AutoGen 偏对话驱动，我的项目是 Pipeline 驱动，LangGraph 更匹配

**Q2：「`send()` 和 `asyncio.gather()` 有什么区别？你为什么不直接用 `asyncio.gather`？」**

**回答要点**：
- `send()` 是 LangGraph 图内并行——子图的结果通过 `operator.add` Reducer 自动归并到全局 State，中间任何一个子图失败不会丢数据
- `asyncio.gather` 是图外并行——你需要手动收集结果、手动合并、手动处理异常
- 本质区别：`send()` 是有状态的图内并行，`asyncio.gather` 是无状态的函数级并行

**Q3：「你的 8 个 Agent 之间是怎么通信的？有没有 Agent-to-Agent 的消息传递？」**

**回答要点**：
- 当前通过共享 TypedDict State 传递数据，不是 Agent 间直接对话
- Intent 节点写入 `competitors` 和 `research_questions` → Planner 读这些字段生成计划 → Profiler 读计划执行采集 → 下游读 profiles 做分析
- 这是 Pipeline 模式的设计选择，不是缺陷。如果要做 Agent 间消息传递，LangGraph 有 `Command` 和 `interrupt` 机制可以支持，这个场景暂时不需要
- 如果面试官追问「那为什么不设计成 Agent 间可以互相提问？」——可以答：竞品情报分析的流程本身就是 Pipeline（采集→分析→报告），不需要 Agent 间动态协商。如果是战略决策支持场景，那需要一个 Central Coordinator 做调度，我预留了扩展空间

**Q4：「Checkpointer 的断点续跑你是怎么测试的？在生产里遇到过什么问题？」**

**回答要点**：
- SQLite 文件锁是多实例部署的瓶颈，当前用 MemorySaver 做默认值、SqliteSaver 是懒加载 fallback
- 单用户场景下 SQLite WAL 模式足够，多用户需要切 PostgreSQL 或 Redis
- 我设计为 `checkpointer` 字段可注入，切换不讲业务代码

---

### 模块二：Plan-and-Execute + Tool Calling

**Q5：「你这个 Plan-and-Execute 和 LangGraph 官方的 `create_react_agent` 有什么区别？为什么不用 ReAct？」**

**回答要点**：
- ReAct 是 Observation → Thought → Action 循环，适合单 Agent 不确定下一步的场景（如代码助手、自主探索）
- 我的 Plan-and-Execute 是「先规划后执行」——Intent Analyst 生成完整的研究计划后，后续 Agent 按计划执行，不需要每步都重新决策
- 竞品分析不需要 ReAct 循环——你知道要搜什么、要抓什么，不需要 Agent "自己决定下一步该干什么"
- 选型理由是「合适的抽象层级」。如果场景换成开放式调研（比如 "帮我研究一个新市场"），ReAct 更合适

**Q6：「三类 Tool 是怎么封装的？如果有新的 Tool 要怎么加？」**

**回答要点**：
- 每个 Tool 独立封装为一个 LangGraph 子图节点，接口统一为 `async def __call__(self, state: CompetitorProfileGraphState) -> dict[str, Any]`
- 新增 Tool 只需三步：1）实现节点函数 2）在 `_build_profile_graph` 中 `add_node` 3）从 `fan_out` 加一条边
- 当前 SearchWorker / ScrapeWorker / RAGRetriever 通过依赖注入支持 mock 替换和 Provider 热切换

**Q7：「Tool 调用失败时的降级策略是什么？失败后会不会影响其他 Tool 的结果？」**

**回答要点**：
- 每个 Tool 独立 `try/except` 包裹，失败返回含 `error: True` 标记的结果，不抛异常
- Aggregator 合并时检查 error 标记，够用的数据继续产出 profile，不够用的维度在报告中标注为 Data Gap
- 三个 Tool 之间完全并行、互不依赖，一个失败不影响另外两个

### 模块三：RAG 与自增长分析记忆

**Q8：「自增长记忆和普通 RAG 有什么区别？RAG 不是本来就是"检索"吗？」**

**回答要点**：
- 普通 RAG 存的是静态文档库（PDF、网页），查一次建一次索引
- 自增长记忆的"增量"在于写路径——每次分析完成后 `rag_ingest` 节点自动将报告/SWOT/市场分析写回 Qdrant，下次查询同赛道时检索到的是历史洞察而非百科词条
- 本质区别：从「知识库」到「记忆系统」。系统运行次数越多，分析质量越高

**Q9：「为什么用 BGE-small-zh 而不是 OpenAI Embedding？」**

**回答要点**：
- 本地运行：零 API 调用成本、零网络延迟、面试现场可演示
- BGE-small-zh 512 维、~100MB 模型文件、CPU 友好，`sentence-transformers` 一行代码加载
- 如果要更好的多语言支持，可以切 `BGE-M3`（1024 维）或 `text-embedding-3-large`——接口是 `Embedder.embed(text) -> list[float]`，一行依赖注入切换

**Q10：「HashEmbedder 和 BGE 的语义差距有多大？你怎么验证切换后确实更好了？」**

**回答要点**：
- HashEmbedder 是用 SHA256 哈希生成伪向量，"SoftBank" 和 "Vision Fund" 在向量空间中完全无关
- BGE 切换后，同一竞品名的 `rag_context` metadata 中的 `report_type` 从 `seed`（静态种子）变为了 `swot` 和 `market_analysis`（历史分析），这是可观测的证据
- 如果需要系统性 benchmark，可以跑一个离线的 retrieval recall@k 对比

### 模块四：Prompt Engineering 与 Structured Output

**Q11：「三层降级的第二层"规则推导"具体怎么做的？和第二层有什么区别？」**

**回答要点**：
- 以 MarketAnalyst 为例：
  - 第一层 `_try_llm_analyze` → DeepSeek 生成结构化市场分析
  - 第二层 `_derived_analysis` → 不用 LLM，从所有 profile 的 `search_results.snippet` 中提取 bigram 频率（停用词过滤 + 大写优先 + min_count≥2），生成 trends；从 profile name 和 summary 推导 competitive_landscape 分类
  - 第三层 `_fallback_analysis` → 纯硬编码模板（"placeholder growth trend"）
- 第二层 vs 第三层的关键差异：第二层仍然产出针对本次查询的个性化内容（只是推导而非 LLM 生成），第三层是通用兜底

**Q12：「LLM-as-Judge 的三维权重 0.4/0.4/0.2 是怎么确定的？有没有做过实验验证？」**

**回答要点**：
- 前两者（完整性、准确性）权重大，因为对于竞品分析来说，数据完整和准确比「这个报告能不能指导决策」更基础——报告准了自然能指导决策
- 没有做过系统性的 A/B 测试，权重是经验设定的。如果需要严格验证，可以用一组标注数据跑相关性分析
- 面试官如果追问"你怎么知道 LLM 评分本身准确"，可以答：这就是 LLM-as-Judge 的固有局限——Judge 也是 LLM，会有偏好。所以 Reviewer 设计为"门控"而非"唯一标准"，规则 fallback 评分提供基准线

**Q13：「json_repair + 自建预处理的修复成功率有多高？你遇到过修不了的情况吗？」**

**回答要点**：
- 自建预处理解决了两种 DeepSeek 高频错误：max_tokens 截断导致的 unterminated string、大 JSON 对象间缺逗号
- 如果 brace-counting 截断后只剩空串，那修不了——此时降级路径启动
- 具体修复成功率没单独统计过，但加了 `import logging; logger.warning(...)`，运行日志中可见 JSON parse failure → fallback 的链路

### 模块五：可观测性与工程实践

**Q14：「`evaluate.py` 的 7 个维度打分逻辑是什么？人工评分对比过吗？」**

**回答要点**：
- 每个维度有明确的评分规则：
  - 竞品真实性：检查竞品名中是否含 "alternative" 或 "替代"，检查是否有对应 source URL
  - Source 可追溯：统计 SWOT evidence 中有多少是真实文本 vs 内部标签（"tavily" 等）
  - 报告可读性：检查 placeholder 计数、executive_summary 是否模板填充、profile summary 是否 "Profile summary for X"
- 纯规则自动评分，没有人工标注对比。它的定位是 CI 质量门控而非学术基准
- 如果需要更高准确度，可以引入 LLM-as-Judge 做语义层面的可读性评分

**Q15：「你说 `_clean_secret` 自动过滤占位 API Key，具体怎么判断一个 key 是占位符还是真实 key？」**

**回答要点**：
- `_clean_secret` 检查 value 中是否包含 `"replace-with"`、`"your_"`、`"your-"` 等占位关键词
- 命中 → 返回 None（相当于未配置），系统走降级路径
- 这是防御性编程的轻量方案——不是密码学安全，但能防止 `.env.example` 的占位值被误当成真实 key 发起 API 调用

### 模块六：架构级追问

**Q16：「如果你要把它改成生产可用的 SaaS 产品，你会做哪些改造？」**

**回答要点**：
1. LLM 层加 retry + circuit breaker（当前单次超时即降级，生产应该重试 2-3 次再降级）
2. Qdrant 从 `:memory:` 切为持久化服务 + 多租户数据隔离
3. Reviewer 门控改为 Human-in-the-Loop：Checkpointer + `interrupt_before` 在发布前暂停，等人工审核
4. 并行子图加并发度控制（当前每个竞品 fork 一个子图，20 个竞品会打爆 Tavily API）
5. 前端从回放模式改为真实时流式：`astream_events` + SSE → UI 增量渲染

**Q17：「你觉得当前架构最大的局限是什么？」**

**回答要点**：
- Pipeline 是静态 DAG——8 节点固定顺序，没有动态路由。如果新增一个数据源类型（如专利搜索），需要改图结构而非热插拔
- Reviewer-Writer 循环有上限（3 次），但没有达到上限后的人工兜底
- RAG 是基础向量检索，没有 Self-RAG 或 Corrective RAG——检索到的内容如果相关度低，没有二次修正机制

## 基于真实面经的模拟面试 Q&A（项目针对性问答）

> 素材来源：7 篇真实 AI Agent 面试面经，按最高频 → 次高频排列。每条回答控制在面试 60 秒内可讲完的篇幅。

---

### 最高频必问（每场面试概率 > 80%）

**Q1：你这个 Agent 系统为什么选 LangGraph？和其他框架（CrewAI、AutoGen）对比过吗？**

LangGraph 给了我三个我需要的底层能力，其他框架没有或者不够好。

第一，显式的状态管理。竞品分析场景下每个 Agent 的输入输出都是 Schema 约束的结构化数据，LangGraph 的 TypedDict State 天然支持字段级别的归并策略——并行子图同时写入 profiles 列表，框架帮你自动合并而不是覆盖。CrewAI 的 Task 抽象没有这个能力。

第二，图内并行。`send()` 让框架为每个竞品 fork 独立子图，单个子图失败不丢成功数据。AutoGen 更偏对话驱动，不适合我这种 Pipeline 场景。

第三，Checkpointer 持久化。graph 跑到任意节点可以暂停、序列化、恢复——这为 Human-in-the-Loop 和断点续跑留了空间。CrewAI 可以做原型，但缺少这个层次的运行时控制。

**Q2：你这个项目是 Plan-and-Execute 还是 ReAct？为什么？**

Plan-and-Execute。竞品分析是确定性流程——你知道要搜什么（竞品名）、要抓什么（网页）、要查什么（历史报告），不需要 Agent 每一步自己决策。Intent Analyst 解析出结构化研究计划后，后续 Agent 严格按计划执行，省掉了 ReAct 每步的 Observation→Thought→Action 循环产生的冗余 LLM 调用。

如果场景换成「帮我探索一个新市场，我不知道里面有什么公司」——那该用 ReAct。合适的抽象层级比分高下更重要。

**Q3：你的 Agent 之间是怎么通信的？有没有 Agent-to-Agent 的直接消息传递？**

当前通过共享 TypedDict State 传递数据，不是 Agent 间直接对话。Intent 写入 competitors 和 research_questions → Planner 读取并生成计划 → Profiler 读取计划执行采集 → 下游 Agent 读取 profiles 做分析。这是 Pipeline 模式的设计选择，不是缺陷——竞品分析本身就是一个采集→分析→报告的线性流程，不需要 Agent 间动态协商。如果场景需要，LangGraph 有 interrupt 和 Command 机制可以支持。

**Q4：从入口到出口，完整描述一遍你的 Pipeline 是怎么跑的？**

用户输入「分析 Notion」，第一站 Intent Analyst 把自然语言解析为目标公司 Notion、市场赛道 SaaS 协作、5 个竞品名和研究问题清单。第二站 Research Planner 为每个竞品生成研究计划。第三站是重头戏——CompetitorProfiler 用扇出机制为 5 个竞品同时 fork 5 个子图，每个子图内 Search、Scrape、RAG 三路并行采集，结果汇聚后生成竞品画像。第四站 MarketAnalyst 汇总所有画像做市场格局分析。第五站 SWOT Synthesizer 为每个竞品生成 SWOT 矩阵。第六站 ReportWriter 组装完整报告。第七站 Reviewer 做质量打分——通过则 RAG 消化记忆，不通过则路由回 Writer 重写，最多 3 次。

**Q5：你的记忆系统是怎么设计的？提取和召回分别怎么做的？**

这和普通 RAG 有本质区别。普通 RAG 存的是静态文档，用一次建一次索引。我的是读写记忆系统——每次分析完成后，Reviewer 审核通过的报告、SWOT、市场分析会自动写回 Qdrant，打上 report_type 和 market_segment 标签。下次查询同赛道时，优先命中历史分析洞察而非初始种子数据。系统运行次数越多，积累的分析经验越多，回答基线越高。召回侧用的是 BGE 语义向量做相似度检索，按 market_segment 过滤。

### 高频追问（概率 50-70%）

**Q6：你怎么防止某类 Agent 把所有活干了，而其他 Agent 闲着？（Supervisor 分配问题）**

我选的是 Pipeline 模式而非 Supervisor 模式，所以不存在分配不均的问题。每个节点职责固定、不可替代——Intent 只管解析、Profiler 只管采集、Reviewer 只管打分。Pipeline 的优势就是确定性，坏处是缺乏灵活性。如果改成 Supervisor 模式，需要引入任务队列 + Agent 能力声明 + 调度策略，我用 Prompt 约束定义每个 Agent 的能力边界来防止越权。

**Q7：Tool 调用的完整流程是怎样的？Agent 是怎么完成一次工具调用的？**

Tool 在我这里是子图节点。Profiler 收到研究计划后，对每个竞品并行触发三个子节点——Search Worker 调 Tavily API 搜索竞品名、Scrape Worker 用 BeautifulSoup 抓取搜索结果中的 URL、RAG Retriever 拿竞品名和市场赛道去 Qdrant 做向量检索。三个 Tool 完全并行、互不依赖，结果通过 Reducer 自动汇聚。每个 Tool 独立做异常隔离——失败了返回 error 标记，不影响另外两个继续跑。

**Q8：为什么选 Qdrant 而不是 Chroma 或 Milvus？**

三个考量。一是零部署成本——`:memory:` 模式开箱即用，不需要 Docker，本地跑和测试都方便。二是 Python 客户端 API 足够简洁，和 LangGraph 的集成没有额外适配成本。三是语义向量对比实验方便——我做了 HashEmbedder 和 BGE 的 A/B 对比，两者在 Qdrant 里就是一个 collection 切换。Chroma 偏轻量但企业级特性弱，Milvus 更重但适合千万级向量，我这个场景几万条就够了。

**Q9：你的 Prompt 怎么设计的？有没有做过上下文工程相关的优化？**

我做了几个方向。第一，每个 Agent 的 Prompt 是结构化模板——先定义输出 Schema（用 Pydantic），再在 Prompt 里明确字段含义和示例，最后让 LLM 输出 JSON 供解析。第二，动态语言适配——自动检测查询是中文还是英文，中文查询以中文输出报告但搜索始终用英文保证召回率。第三，Rule-based Reviewer 和 LLM-as-Judge 双轨并行——前者给后者提供评分基准线，防止 LLM 评分漂移。第四，三层降级里第二层规则推导是从真实数据中统计提取而不是编造，这个设计保证了降级产出的内容仍然和本次查询相关。

**Q10：三层降级里第二层「规则推导」具体是怎么做的？和第二层、第三层有什么区别？**

以 MarketAnalyst 为例。第一层 LLM 生成结构化市场分析。第二层不用 LLM——从所有 profile 的搜索摘要中提取高频 bigram（停用词过滤、大写优先），生成 trends；根据竞品定位推导 competitive_landscape 分类。第二层和第三层的关键差异：第二层仍然产出针对本次查询的个性化内容（只是推导而非生成），第三层是通用兜底模板。所以系统在没有 API Key 的情况下也能给出有意义的结果。

### 中频追问（概率 30-50%）

**Q11：Agent 系统的整体效果怎么评估？出现 Badcase 怎么定位到哪个环节出了问题？**

我自建了 7 维度的评估框架，纯规则评分，每一维度有明确的评分逻辑。竞品真实性查竞品名里有没有「alternative」这种幻觉标记、有没有对应来源 URL。Source 可追溯性统计 SWOT evidence 里真实抓取文本和内部标签的占比。降级行为查报告里 placeholder 出现次数和模板句式。

Badcase 定位靠 ExecutionTracker——每个节点执行完都会记录 checkpoint（哪个节点、什么状态、什么输出），JSONL 持久化了可以直接回溯。报告质量差先去 Reviewer 的评分明细，三维分项打分能告诉你问题是内容不完整还是结论不准确，然后顺藤摸瓜找上游的搜索或抓取数据。

**Q12：Prompt 调优中遇到过「修好一类坏了另一类」的问题吗？你怎么解决的？**

用 Reviewer 的质量门控作为反馈信号。每次修改 Prompt 后跑同一个 query，看三维评分的变化——如果可操作性涨了但准确性跌了，说明新 Prompt 让 LLM 更发散但离事实更远，需要收紧约束。另外我保留了 31 个种子报告做冷启动基线，新 Prompt 效果可以和基线对比。

**Q13：什么时候该调 Prompt，什么时候该做微调？**

Prompt 先行——成本低、可逆、可解释。当 Prompt 调到极限（改了不改效果都一样差），且你在同一类问题上反复踩坑，才考虑微调。但如果问题是「数据不完整」而不是「模型理解错」，微调没用——需要补的是搜索覆盖率和 RAG 数据质量，不是模型参数。

**Q14：你的 Tool 是怎么封装的？如果要加一个新 Tool 怎么做？**

每个 Tool 统一封装为子图节点，接口一致——接收 State 返回 dict。新增 Tool 三步：实现节点函数、在子图里 add_node、从扇出点加一条边。不改已有代码。三类 Tool 就共享了同一个容错逻辑（独立异常隔离 + error 标记），新增的 Tool 自动继承。

**Q15：你的项目是 Workflow 还是真正意义上的 Agent？为什么不做全自主 Agent？**

当前是 Workflow-based Agent——固定 DAG 编排。竞品分析场景下，流程是确定性的，Workflow 提供了可预测性和可审计性，这两个在商业场景中比灵活性更重要。全自主 Agent 在开放域任务（比如「研究一个新市场」）更有优势，但垂域竞品分析追求的是稳定输出质量而非探索能力。而且我在某些环节（Reviewer 评分 + 条件路由）已经引入了 Agentic 的决策机制——系统会根据评分自动决定通过还是重写。

**Q16：你这个项目在架构上最大的局限是什么？如果让你重新设计会怎么做？**

三个局限。第一，Pipeline 是静态 DAG——新增数据源类型需要改图结构，理想方案是 Tool 注册表 + 动态子图组装，新 Tool 热插拔。第二，RAG 是基础向量检索——检索到的内容如果相关度低没有二次修正机制，演进方向是 Self-RAG 或 Corrective RAG，检索后加一步 LLM 做相关性评分决定是否改写 query 重查。第三，Reviewer-Writer 循环有 3 次硬上限——达到上限后没有人工兜底。这三个我在简历里也写了。

### 低频但加分（概率 < 30%）

**Q17：你对 Context Engineering 怎么理解？在这个项目里有什么体现？**

Context Engineering 是管理模型能看到的上下文内容和结构——分阶段注入、按需裁剪、防污染。在这个项目里体现为：每个 Agent 不是拿到一整段 State，而是从全局 State 中按需拿字段。Profiler 只需要 competitors 和 research_questions，不需要看 Reviewer 的评分历史。降级场景中，第二层规则推导基于真实数据统计而不是凭空编造，本质上是确保降级上下文仍然包含与 query 相关的事实信息。

**Q18：如果让你把这个项目做成生产可用的 SaaS 产品，你会做哪些改造？**

五件事。LLM 层加重试和熔断（当前单次超时就降级，生产应该重试 2-3 次）。Qdrant 从内存模式切换到持久化服务加多租户数据隔离。Reviewer 门控改为 Human-in-the-Loop 审核（Checkpointer + 发布前暂停）。并行子图加并发度控制（当前每竞品 fork 一个子图，20 个竞品会打爆搜索 API）。前端从回放模式改为实时流式（SSE + UI 增量渲染）。

**Q19：你对 Agent 的 Skill 机制怎么理解？**

Skill 是预定义的、可被 Agent 按需调用的能力模块——和 Tool 的区别在于 Skill 不仅是执行一个 API，而是携带了使用场景、约束和回退策略的知识。如果给我的系统加 Skill 机制，我会把「竞品筛选」「SWOT 生成」「来源可信度校验」封装为可插拔 Skill，每个 Skill 自带 Prompt 模板、容错策略和输出 Schema，让 Agent 按任务复杂度动态加载而不是全量注入。

## RAG 技术细节面试防守（基于真实面经高频追问）

> 以下是面经中 RAG 相关的高频 / 中频问题，基于项目实际代码逐条编写。每条回答控制在 60 秒口述篇幅，可独立使用。

---

### 一、基本概念与选型（每场必问）

**Q1：你这个 RAG 系统整体是怎么设计的？**

分三层。**Embedding 层**——定义了一个协议接口 `embed(text) -> list[float]`，下面挂了两个实现：BGE-small-zh 语义向量（512 维）和 HashEmbedder 轻量向量（64 维），环境变量一行切换。**存储层**——Qdrant，支持 `:memory:` 模式和磁盘持久化两种运行方式，距离度量用 cosine。**业务层**——检索的 query 是 `市场赛道 + 竞品名` 拼起来的，不是单纯的竞品名，这样能召回同赛道的相关分析而不仅是名字精确匹配。

**Q2：为什么选 Qdrant 而不是 Chroma 或 Milvus？**

三个考量。一是零部署成本——`:memory:` 模式开箱即用，不需要 Docker，本地开发和测试都方便。二是 Python SDK 简洁，跟 LangGraph 集成没有额外适配。三是我做了两种 Embedding 的 A/B 对比，Qdrant 里切换 collection 就行，两者体验一致。Chroma 偏轻量但企业级特性弱，Milvus 更重更适合千万级以上向量——我这个场景几万条就够了。

**Q3：为什么用 BGE-small-zh 而不是 OpenAI Embedding？**

本地运行——零 API 调用成本、零网络延迟，面试现场可以演示。模型文件只有 ~100MB，CPU 推理，`sentence-transformers` 一行代码加载。如果要更好的多语言支持可以换 BGE-M3（1024 维）或 text-embedding-3-large——因为 Embedding 层是协议抽象的，切模型就是换一个实现类，不改业务代码。

---

### 二、数据处理与分块（高频追问）

**Q4：你是怎么做文本切分的？chunk 大小和重叠怎么定的？**

固定大小切分，chunk_size=900 字符，overlap=120 字符。切之前先做 whitespace 归一化（多空格合并），短于 900 的不切。为什么是 900——竞品报告段落通常在 500-1000 字之间，900 能保证一个 chunk 包含一个完整的分析段落而不是半截句子。120 的重叠是为了保证 chunk 边界处的关键信息不会因为刚好切断而丢失。点 ID 是 SHA-256 哈希生成的确定值（`source:chunk_index:chunk` 拼起来取前 8 字节转 int），相同内容重新摄入不会产生重复——是 upsert 而不是 insert。

**Q5：如果 chunk 切分丢了上下文关联性怎么办？有没有了解过 GraphRAG？**

了解过。当前固定大小切分确实会丢跨段落的语义关联——比如 SWOT 的 Strength 和对应的 evidence 可能被切成两个 chunk。GraphRAG 的思路是用实体和关系构建知识图谱，检索时不仅召回相似文本，还能沿着关系边找到关联信息。如果让我改进，两步走：第一步先把竞品名、SWOT 维度、市场赛道作为节点，报告段落作为边属性构建轻量图；第二步检索时先从向量库召回 top-k chunk，再从图里取相邻节点补充上下文。

**Q6：RAG 检索的 query 是怎么构造的？做过查询改写吗？**

query 是 `市场赛道 + 竞品名` 拼的，比如「SaaS 协作 Notion」。这样设计是因为如果只搜竞品名，只能召回该竞品的历史分析；加上市场赛道后，同赛道的其他竞品分析也会被召回——比如搜 Notion 能命中之前 Notion vs Coda 的对比分析。

目前没有做查询改写。如果竞品名有歧义（比如 Apple 到底是科技公司还是水果？），需要加一层改写——用 LLM 把 query 扩展为「Apple Inc technology company competitive analysis」再加权重排序。

### 三、自增长记忆闭环（项目核心亮点，面试官必追问）

**Q7：「自增长记忆」和普通 RAG 到底有什么区别？RAG 不就是检索吗？**

普通 RAG 是只读知识库——存静态文档，查一次建一次索引。我的 RAG 是读写记忆系统——每次分析完成后，Reviewer 审核通过的报告、SWOT、市场分析会自动写回 Qdrant，打上 metadata 标记（report_type=swot/market_analysis/executive_summary）。下次查同赛道时，优先命中历史分析洞察而非初始种子数据。

本质差异：存储内容会随着使用增长，且增长的内容是经过质量门控筛选的（只有 Reviewer 通过的报告才写回）。系统运行次数越多，积累的分析经验越多，回答基线越高。面试时可以一句话总结：「从知识库升级为记忆系统」。

**Q8：写回的时候具体写什么内容？怎么避免重复写入？**

写回 5 类内容：执行摘要（executive_summary）、每个竞品的 SWOT 分析、市场分析（market_analysis）、以及每个竞品的 profile summary。每类打上不同的 report_type 元数据标签。

去重用确定性的点 ID——对 `source:chunk_index:chunk` 做 SHA-256 取前 8 字节转整数。相同内容重新写入时 ID 相同，Qdrant 的 upsert 操作会自动覆盖而非追加，不会产生重复数据。

**Q9：历史分析和种子数据怎么区分？二次查询时怎么优先用历史数据？**

每条数据写入时带 metadata，其中 report_type 字段标记来源：`seed` 是冷启动种子、`swot` / `market_analysis` / `executive_summary` 是历史分析沉淀。检索结果是按 cosine 相似度排序的，对同赛道查询，包含具体分析内容的历史数据相似度自然高于通用种子数据，不需要额外做优先级逻辑。

我的评估框架里专门有一维度「RAG 记忆利用」来量化这个效果——统计返回结果中历史数据 vs 种子数据的占比，验证自增长机制是否真正生效。

**Q10：种子数据是什么？为什么需要？**

31 个种子竞品报告，覆盖四大赛道：SaaS 协作（Notion、Slack 等）、VC 投资（Sequoia、a16z 等）、大厂 AI（Google、OpenAI 等）、消费硬件（Tesla、BYD 等）。内容是占位的（一行简介），目的是让向量库冷启动时有数据可查，而不是空库。第一次分析完成后种子就会被历史分析逐步替代。

---

### 四、Embedding 可插拔设计（面试官在乎抽象能力）

**Q11：你的 Embedding 层怎么做到可插拔的？**

定义了一个 Protocol 协议 `Embedder`，只要求两个东西：`dimensions` 属性和 `embed(text) -> list[float]` 方法。然后两个实现类——HashEmbedder（SHA-256 哈希生成 64 维伪向量，确定性、零模型依赖）和 SentenceTransformerEmbedder（加载 BGE 模型，512 维语义向量，延迟加载第一个 embed 调用时才初始化模型）。切换方式——环境变量 `EMBEDDING_MODEL` 填模型名就用 BGE，留空就用 HashEmbedder。业务代码只依赖协议不依赖具体类，一个工厂函数自动解析。

**Q12：HashEmbedder 是什么？为什么设计了这么个东西？**

轻量级的确定性伪向量——对文本分词后每个 token 做 SHA-256，用哈希值决定在 64 维向量中的位置和正负方向。它不是语义向量（「SoftBank」和「Vision Fund」对它来说是无关的），但有两个好处：一是零依赖、零模型加载，pip install 完直接跑，适合测试和 demo；二是完全确定性，同一输入永远同一向量，方便 CI 验证。真实场景用 BGE，开发和测试用 HashEmbedder。

**Q13：你怎么验证 BGE 确实比 HashEmbedder 好？**

两个指标。一是语义相似度——`cosine(SoftBank, 红杉资本)=0.820`（两个投资机构，BGE 认为相关）vs `cosine(SoftBank, 天气)=0.677`（语义无关，得分低），HashEmbedder 两者都是随机值。二是检索相关率——同一 query 检索 Top-5 结果，HashEmbedder 模式下全部是种子数据（0/5 命中相关历史分析），BGE 模式下 5/5 全部命中相关历史分析或同赛道种子数据。

---

### 五、RAG 质量评估（高频追问）

**Q14：你怎么评估 RAG 系统的质量？有没有做数据集验证？**

我的评估框架里有专门两个维度评估 RAG。「RAG 记忆利用」维度统计检索结果中历史分析 vs 种子数据的占比，量化自增长机制是否生效。「Source 可追溯」维度检查 SWOT 分析里的 evidence 是真实抓取文本还是内部标签，间接验证 RAG 提供的上下文是否被下游 Agent 真正使用。

目前没有构建专门的 RAG 评估数据集。如果需要，可以做一个离线的 recall@k 对比——人工标注一批查询的正确答案，比较不同 Embedding 模型、不同分块策略的检索命中率。

**Q15：怎么防止 RAG 检索到的内容被 LLM 直接用但不准确？**

两层防护。第一，RAG 给 Agent 的是上下文参考，不是唯一答案——Agent 会把 RAG 结果和实时搜索、网页抓取的结果交叉验证。只有一个信息源的结果在报告中标注为低置信度。第二，Reviewer 的三维评分里「准确性」占 0.4 权重——如果 LLM 基于不准的 RAG 数据写出了错误结论，Reviewer 会给出低分并退回重写。本质上 RAG 是记忆辅助，不是唯一真理来源。

### 六、进阶追问（加分项）

**Q16：和 GraphRAG 比，你的方案有什么局限？**

GraphRAG 的优势是能处理跨段落的实体关系——比如「Notion 的弱点是定价」和「Airtable 的定价更灵活」这两段在不同 chunk 里，向量检索可能只召回一段，但 GraphRAG 可以通过「定价」这个实体把两端连起来。我的当前方案是纯向量检索，跨 chunk 的关联靠 overlap 和 query 构造来弥补，但效果不如显式构图。如果竞品分析场景需要跨竞品对比推理，GraphRAG 会是更好的选择。

**Q17：如果 RAG 检索到的内容完全不相关怎么办？有没有二次检索机制？**

当前没有 Self-RAG 或 Corrective RAG 机制——检索结果直接给 Agent 用，不做二次相关性判断。如果引入，方案是检索后加一步 LLM 相关性评分（0-10），低于 5 分的 chunk 丢弃，然后自动改写 query 做第二轮检索。我的「架构自省」里已经标注了这个改进方向。
