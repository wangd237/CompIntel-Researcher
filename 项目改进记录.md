> 基于实际运行日志，记录 CompIntel 当前问题及改进方向。

## 1. LLM 超时与 JSON 截断问题

**现象**：运行 `分析 Notion 在协作工具市场的主要竞品` 时，report_writer 反复出现 LLM 超时或返回破损 JSON，触发降级路径。

**日志证据**：

```
Failed to parse JSON candidate ─ Unterminated string / Expecting ',' delimiter（多次，截断位置不同）
LLM call failed, returning None ─ RuntimeError: The read operation timed out
```

**根因**：四重因素叠加——

| 因素 | CompIntel 当前值 | gpt-researcher 做法 |
|------|:-----------:|------|
| 单次 LLM 调用体量 | 5 个竞品 + SWOT + 市场分析 + sources 全塞进一个 prompt | 按子主题拆开，每个独立调用 |
| 输出格式 | 嵌套 Pydantic JSON（一个逗号坏了整体作废） | 自由格式 Markdown（截断可读） |
| max_tokens | 2600 | 6000（且只用在一个子主题上） |
| 重试机制 | 无，超时即降级 | 10 次重试 + 指数退避（1s→2s→4s→8s） |

**结论**：不是 DeepSeek 不行，是**单次调用压力过大 + JSON 容错空间为零 + 无重试**共同导致降级成为常态而非异常。

---

### 架构反思：为什么参数调优不是根本解

直观的修复思路是加超时、加 token、加重试——但这是把症状压下去，不解决结构问题。真正的架构缺陷在于：**ReportWriter 是一个单体调用**——所有上游数据灌入一个 Prompt，让 LLM 一次性输出完整嵌套 JSON。

这个设计的**硬天花板**是：

1. **竞品数量不可扩展**。5 个竞品 Prompt 已经很大，10 个竞品必崩。每个竞品 profile 包含 search_results、scraped_content、rag_context，Prompt 随竞品数线性膨胀。
2. **JSON 嵌套深度 × LLM 截断 = 故障放大**。报告 JSON 有三层嵌套（report → sections[] → key_insights[]），LLM 在任何一层截断都会导致整份 JSON 无法解析。不是「丢了最后一段」——是「整份报告没了」。
3. **重试治标不治本**。同一个大 Prompt 重试 3 次，每次超时概率相近。重试解决的是网络抖动，不是「任务本身超出 LLM 单次处理能力」。

---

### 架构方案：Section-by-Section 装配（借鉴 gpt-researcher 的子主题拆分）

核心思想：**不让 LLM 一次性生成整份报告。拆成多个小调用，每个调用只做一件事，最后用代码装配。**

#### 改前 vs 改后

```
改前（单体调用）：
  _try_llm_write(5 profiles + SWOT + market + sources)
    └─ 1 次 LLM 调用 → max_tokens=2600 → 嵌套 JSON
    └─ 失败 → 整份报告降至 derived/fallback

改后（分段装配）：
  _generate_report_assembly(state)
    ├─ _generate_executive_summary(market_analysis)
    │     prompt ~400 tokens → output ~200 tokens, timeout=30s
    │     失败 → 模板填充（已有 _derived 逻辑）
    │
    ├─ _generate_competitor_narratives(profiles)  ← 并行
    │     ├─ competitor_1: prompt ~300 tokens → max 400 tokens
    │     ├─ competitor_2: prompt ~300 tokens → max 400 tokens
    │     └─ competitor_N: prompt ~300 tokens → max 400 tokens
    │     单段失败 → 用 profile.summary + key data points 填充
    │
    ├─ _generate_conclusion(query, narrative_summaries, swot)
    │     prompt ~600 tokens → output ~300 tokens, timeout=30s
    │     失败 → 基于搜索/抓取统计数据推导（已有 _derived 逻辑）
    │
    └─ _assemble_report(exec_summary, narratives, market, swot, conclusion, sources)
          纯代码组装，无 LLM 调用
          产出 Pydantic Schema（下游接口不变）
```

#### 为什么这个方案解决了根因

| 维度 | 改前 | 改后 |
|------|------|------|
| 每次调用输入 | 全部 5 竞品数据 (~3000+ tokens) | 单个竞品或单个维度 (~300-600 tokens) |
| 每次调用输出 | 完整嵌套 JSON (max 2600 tokens) | 扁平 dict 或纯文本 (max 400 tokens) |
| 截断后果 | 整份 JSON 作废 | 只丢一个段落，其余完整 |
| 超时概率 | 高（2600 token 结构 JSON > 60s） | 低（400 token 文本 ~8-15s） |
| 故障粒度 | 整份报告降级 | 单段降级，其他段 LLM 质量保留 |
| 竞品扩展 | 5 个是软上限 | 每增一个只是多一次独立调用 |
| 并行能力 | 无 | 竞品叙事段可并行生成 |

#### 关键设计决策

**1. 竞品叙事段用纯文本而非 JSON**

每个竞品段只要求 LLM 输出 2-3 段 Markdown。不做 JSON 解析——截断了只是最后一句丢了，其余内容可用。从 Markdown 提取 title、content、key_insights 是确定的正则逻辑，不依赖 LLM 格式。

**2. 装配层是纯代码，不调 LLM**

`_assemble_report` 把所有分段拼成最终的 Pydantic Schema。这个函数没有网络调用、不依赖 LLM、不会失败。它做三件事：排列 section 顺序、补 source 引用、汇总 data_gaps。

**3. 每段独立降级，互不影响**

不再是「一个 LLM 调用失败 → 全部走 derived」。如果 conclusion 段生成失败而 narratives 全部成功，最终报告有 5 段 LLM 质量的竞品分析 + 1 段 derived 结论——远好于全部 derived。

**4. Reviewer 接口不变**

装配层出品仍然是 `report: dict[str, Any]`，包含 `title / executive_summary / sections / conclusion / sources / data_gaps`。Reviewer 不做任何改动。

#### 实施复杂度

| 改动 | 文件 | 量 |
|------|------|:--:|
| 新增分段生成方法 | `report_writer.py` | ~120 行 |
| 新增装配方法 | `report_writer.py` | ~50 行 |
| 重构 `__call__` 调度逻辑 | `report_writer.py` | ~30 行 |
| 保留现有降级全套逻辑 | 不改动 | 0 |
| 并行生成（`asyncio.gather`） | `report_writer.py` | ~10 行 |
| 新增测试 | `test_compintel_core.py` | ~30 行 |

总计约 240 行新增/改动，Reviewer 和下游消费者零改动。

#### 与「加参数」方案的对比

| | 参数调优（shallow fix） | 分段装配（architectural fix） |
|------|:---:|:---:|
| 改动量 | ~10 行 | ~240 行 |
| 解决当前问题 | 部分（5 竞品可能缓解，10 竞品复发） | 彻底 |
| 解决扩展性 | 否 | 是 |
| 引入新复杂度 | 否 | 架构复杂度略有提升 |
| 可测试性 | 不变 | 提升（每段可独立 mock 测试） |

**建议**：两步走。第一阶段参数调优（10 行，当天完成，缓解当前问题）；第二阶段分段装配（240 行，正确长期解）。第一阶段不是浪费时间——它让你在第二阶段开发期间系统的降级路径继续工作。

## 2. JSON 解析容错评估

**现状**：3 层修复都做了（`json_repair` → brace-counting 截断修复 → unterminated string 闭合），但 1 处失败全部丢弃。

**改进方向**：

1. `load_repaired_json` 失败时，不返回 `None`，返回 `{"raw": text, "parse_error": True}`，让 Agent 尝试从原始文本中正则提取可用的顶层字段
2. 加结构化输出的 fallback——如果 JSON 解析失败，退回到 Markdown 解析路径（正则提取标题、段落、列表）
3. 记录 JSON 修复成功率（`logger.warning` 已有，但没聚合），为后续 Prompt 调优提供反馈信号

## 3. gpt-researcher 可借鉴的设计

| 设计 | 说明 | CompIntel 采纳优先级 |
|------|------|:---:|
| 子主题拆分 + 逐节生成 | 大报告拆成 N 个小 Prompt，每次只生成一节的 Markdown | 高 |
| 纯文本输出 + 后解析 | 先生成 Markdown，再用正则提取结构化信息（而非要求 LLM 直接出 JSON） | 高 |
| 指数退避重试 | 失败后 1s→2s→4s→8s 重试，最多 10 次 | 高 |
| 降级 Prompt（去 system prompt 重试） | 第一次失败后用更简单的 Prompt 重试一次才彻底放弃 | 中 |
| 流式输出 + WebSocket 推送 | `stream=True`，前端实时看到生成中的内容 | 低（当前是回放模式） |

### 7.3 架构异味（待重构）

#### 重复代码（粘帖复制）

| 模式 | 出现位置 | 次数 |
|------|---------|:--:|
| `completion_fn` 懒加载 + import 块 | research_planner, market_analyst, swot, report_writer, reviewer | 6 |
| `provider, model = _split_provider_model(settings.xxx)` | 每个 LLM 调用方法开头 | 6 |
| `try: await fn(messages=[...]) except TypeError: await fn(prompt)` | 4 个 agent | 4 |
| list 标准化 `_normalize_list` / `_normalize_list_field` / `_normalize_list_of_strings` | market_analyst, report_writer | 3 |
| 三级降级 `llm_key? → try LLM → derived → template` | market_analyst, swot, report_writer | 3 |

#### 缺失的抽象

1. **`BaseCompIntelAgent` 贫血基类** — 只存了 `model` 字符串，没有共享 LLM 调用/重试/解析/降级逻辑，零减负
2. **没有 LLM Service 层** — 没有封装 `import → 调用 → 重试 → 解析` 的统一入口，每个 agent 自己重复造
3. **没有 State Adapter** — 每个 agent 开头重复 `if isinstance(state, dict): field = state.get("field") or {}`
4. **API key 命名不一致** — IntentAnalyst 用 `openai_api_key`，其他 agent 用 `llm_api_key`，SearchWorker 用 `search_api_key`

#### 分层过载

5. **`graph.py` 539 行** — 混杂编排(graph build) + 数据处理(profile summary) + RAG 生命周期(bootstrap/ingest) + 语言检测 + source 提取，四种职责
6. **`report_writer.py` 804 行** — 一个类包含了段落生成、文本提取、拼装器、回退报告、引用格式化、截断逻辑、语言模板

#### 状态管理问题

7. **`state.py` 67 字段中 14 个从未被读写** — `status`/`phase`/`blockers`/`metadata` 等是设计好但从未实现的跟踪系统
8. **`CompetitorProfilerState` 定义后从未导入** — 死代码
9. **子图双层嵌套输出** — worker 返回 `{"search_results": {..., "search_results": [...]}}`，aggregator 必须手动 unwrap

### 7.4 目标干净架构

```
compintel/
  core/
    llm_service.py          # 封装 import → 调用 → 重试 → 解析，统一入口
    state.py                # 精简到只有实际使用的字段
    state_adapter.py        # 类型化 state 读写，消除 isinstance 检查
  agents/
    base.py                 # 注入 llm_service + state_adapter，共享降级链
    intent_analyst.py       # prompt + schema，其余由 base 处理
    planner.py
    profiler.py             # 直接调 workers（删掉子图），asyncio.gather
    workers/
      search.py
      scrape.py
      rag.py
    market_analyst.py       # 共享 mixin 实现三级降级
    swot.py
    report_writer/
      executive_summary.py
      competitor_narrative.py
      conclusion.py
      assembler.py
      fallback.py
    reviewer.py
  graph.py                  # 只做 LangGraph 拼装 + 路由 + fan-out（目标 <200 行）
```

**核心原则**：每个 agent 只写 prompt 和 output schema；LLM 调用/重试/JSON 解析/三级降级全部由 base/core 统一处理。

## 9. 高并发问题审计（2026-06-23）

> PM 分析了系统的并发架构后，发现系统缺乏全局并发控制，在 DeepSeek free tier 限流约束下会自我击溃。以下仅记录问题与根因，暂不修复。

### 9.1 并发放大链

```
query → intent_analyst → research_planner
  → [fan-out: N × competitor_profiler]       ← N 由 LLM 输出，无上限
    → 每个 profiler 内 3 并行分支:
        search_worker | scrape_worker | rag_retriever
  → market_analyst → swot_synthesizer → report_writer → reviewer
  → approved? → END
  → revise?   → loop 最多 3 轮（重新走 fan-out）
```

**放大倍数**：N=5 竞品时，profiler 阶段同时发出 **15 个并行 HTTP 请求**（5×3），每个失败重试 3 次 → 峰值 **45 个请求堆积**。若 reviewer 打回 3 轮 → 全部 ×3。

### 9.2 问题清单

#### P0 — 阻断级（不修则系统不可用）

| # | 问题 | 根因 |
|---|------|------|
| G1 | **全局 LLM 并发无上限**。15 个并行请求同时打向 DeepSeek free tier，触发 SSL handshake timeout / IncompleteRead，流水线崩溃，用户零结果。 | 系统没有任何全局并发控制机制。`report_writer` 和 `scrape_worker` 内部虽有局部 Semaphore(3)，但互不感知。profiler fan-out 阶段完全无节制。 |
| G2 | **HTTP 连接无复用**。每个 LLM 请求新建 TCP 连接 + TLS 握手（耗时 200-500ms），用完即丢。并发时大量短命连接可能触发本地端口耗尽。 | 所有 HTTP 调用使用 `urllib.urlopen` 包在 `asyncio.to_thread` 中，无连接池、无 keepalive、无 `aiohttp`/`httpx`。 |
| G3 | **Fan-out 无上限**。`graph.py:240-254` 的 `_fan_out_competitors` 按 `len(competitors)` 动态分支，无最大限制。LLM 若返回 10 个竞品名，瞬间 30 个并行请求秒崩。 | `Send` fan-out 是 LangGraph 原生能力，但代码中未加 `min(N, MAX)` 硬限制。LLM 输出不可控，必须防御。 |

#### P1 — 高优（严重浪费资源）

| # | 问题 | 根因 |
|---|------|------|
| R1 | **重试无退避，制造二次洪峰**。第一次请求因限流失败后，所有调用在同一时刻重试，再次全部失败。3 轮重试 = 3 次集中洪峰，最终大概率仍然失败。 | `call_and_parse` / `call_with_reasoning` 虽有指数退避（1s→2s→4s），但各调用之间没有错峰抖动（jitter），并发调用仍然同时重试。 |
| R2 | **Reasoner 空响应浪费 token**。`deepseek-reasoner` 或 `deepseek-v4-pro` 等推理模型把全部 token 烧在链式思考上，`content` 为空，`reasoning_content` 中也没有 JSON，触发重试后再白扣一次 token。 | DeepSeek 推理模型（reasoner / v4-pro）的响应结构中 `content` 可能为空字符串，`reasoning_content` 包含完整思考过程但不含 JSON。当前 `_post_chat_completion` 中的 `_extract_tail_json` 在思考文字中找不到 JSON 时直接抛异常 → 触发重试 → 再扣 token。 |
| R3 | **无自感知限速**。系统反复撞限流墙，每次都报同样的错，从不调整行为。用户觉得"这工具怎么每次都犯同样的错"。 | DeepSeek free tier 的速率限制阈值未知且不可配。系统没有"遇到 429 → 降低并发 → 试探恢复"的自适应机制。 |

#### P2 — 待办（改进鲁棒性）

| # | 问题 | 根因 |
|---|------|------|
| C1 | **Profiler 内部 3 路并行无节制**。search_worker、scrape_worker、rag_retriever 三个分支同时抢占全局配额，互不协调。 | 子图 fan-out 三路并发设计上是为了加速单个竞品分析，但未与全局并发控制联动。 |
| C2 | **Revise 循环内重复放大**。reviewer 打回一次，整个流水线（含 fan-out）全部重跑。打回 3 次 = 跑了 4 遍竞品画像。 | `graph.py` 的 revise 循环粒度是整个 `report_writer → reviewer` 段，但竞品画像（profiler）的上游结果没有在 revise 中复用。 |
| C3 | **LLM 调用无超时保护**。偶尔调用卡死不报错不返回，占住并发槽位，其他调用排队等待。 | `llm_timeout_seconds` 默认 90s，但 `urlopen` 的超时行为取决于底层 TCP 栈，某些情况下可能更久。部分调用路径可覆盖 timeout，但没有强制全局硬超时。 |

#### P3 — 远期（体验优化）

| # | 问题 | 根因 |
|---|------|------|
| E1 | **无进度可见性**。运行 10 分钟，终端只显示几条日志。用户不知道在干什么、完成了多少、还要多久。 | 流水线以异步 fan-out 方式执行，当前只有 `execution_log` 事件汇总在最终报告里，运行时无实时进度输出。 |
| E2 | **无优雅降级**。部分竞品分析失败时，整个流水线中断，用户什么输出都拿不到。 | `_profile_one_node` 虽已有 try/except 兜底，但下游节点（market_analyst、report_writer）在缺少数据时只能产出占位内容或直接失败。 |
| E3 | **无响应缓存**。重试时相同 prompt 重复调用 LLM，浪费 token 和时间。reviewer revise 循环中也重复调用。 | 无请求级缓存。`temperature>0` 时不应缓存，但 `temperature=0` 的格式化调用理论上可安全缓存。 |

## 10. 爬取信源权威性与内容质量改善（2026-06-24）

> 用户反馈：SWOT 四象限中 W/T/O 频繁出现 "no evidence"；竞品对比板块内容混乱；最终报告质量差。根因溯源：数据端、SWOT 端、报告端三重缺陷叠加。

### 10.1 问题链路

```
搜索质量差 / 爬取失败 / RAG 空
        │
        ▼
  profile 数据贫瘠（summary 只有一句话，search_results 是占位符）
        │
        ├──→ SWOT: LLM 拿不到 W/T/O 的证据 → 填 "no evidence"
        │         （5 个竞品 × 4 象限 = 20 个格子挤在 2200 token 里）
        │
        ├──→ 竞品对比: market_analyst 给分类列表 +
        │         SWOT 给 cross_analysis +
        │         report_writer 各写各的 narrative
        │         → 三份内容互相不引用、格式不统一、术语不协调
        │
        └──→ 报告: executive_summary 先写 → competitor narratives → conclusion
                  ↑ 总结写在最前面，它还没看过后面每个 section 的实际内容
                 各 section 独立生成，没有第二遍统稿
```

### 10.2 爬取信源的三重缺陷

| # | 缺陷 | 根因 | 对 SWOT 的影响 |
|---|------|------|------|
| 1 | **竞品官网是 PR 稿** | `_build_target_urls()` 默认爬竞品自家网站的首页/定价/关于。小米官网不会说自己的车充电慢，瑞幸官网不会说自己亏损。 | 爬出 S/O 数据多、W/T 数据几乎为零 → W 和 T 天然没 evidence |
| 2 | **行业站点覆盖太窄** | `_INDUSTRY_SCRAPE_SOURCES` 仅覆盖汽车、新能源、SaaS、投资 4 个行业。咖啡/消费品/医药等行业直接掉进 G2/Capterra 兜底——两个软件评测站搜"瑞幸咖啡" → 空结果 → profile 空洞 | 非 4 个行业的所有查询都无法获得行业特化的信源 |
| 3 | **信源用途未区分** | 系统不去能产生 W/T 证据的地方找。投诉平台、财报风险披露、知乎批评文章、用户差评——这些是 W/T 的天然来源，但当前 zero coverage。 | S/O 能拿到 evidence，W/T 只能编造或空置 |

### 10.3 信源应区分用途

不同信息需要去不同的地方找：

| 要找什么 | 应该去哪 | 当前状态 |
|------|------|:--:|
| 竞品基本面（定位、产品线、规模） | 官网 + 企查查 | ✅ 官网 |
| 正面评价（S/O 来源） | 行业评测站、用户好评 | ✅ 行业站 |
| 负面信息（W/T 来源） | 投诉平台（黑猫/12315）、财报风险披露、知乎/小红书批评 | ❌ 完全没有 |
| 客观数据（营收、份额、增长） | 财报、招股书、行业报告 | ❌ 偶然碰到 |
| 用户口碑（体验、定价感知） | 社交媒体、论坛、应用商店评论 | ❌ 完全没有 |

### 10.4 分行业权威信源白名单（建议）

| 行业 | 正面/评测信源 | 负面/风险信源 | 数据信源 |
|------|------|------|------|
| 汽车/新能源 | 汽车之家、懂车帝、insideEVs、42号车库 | 车质网、黑猫投诉、汽车之家论坛 | 乘联会、工信部目录 |
| 餐饮/咖啡 | 窄门餐眼、FoodTalks、咖门 | 黑猫投诉、知乎、美团差评 | 企查查、艾媒咨询 |
| 消费电子 | 爱范儿、少数派、Zealer | 京东差评、黑猫投诉、B站测评 | IDC、Counterpoint |
| SaaS/软件 | G2、Capterra、少数派 | Product Hunt 低分、Trustpilot 差评 | 年报/招股书 |
| 医药/生物 | 医药魔方、Insight、动脉网 | FDA 警告信、药监局飞检 | 临床数据、年报 |
| 金融/投资 | 雪球、同花顺 | 证监会处罚、黑猫投诉 | 财报、Crunchbase、PitchBook |
| 电商/零售 | 亿邦动力、36氪 | 黑猫投诉、社媒吐槽 | 企查查、晚点LatePost |

### 10.5 建议方案

**核心原则**：数据源不应硬编码在代码中，而应可配置（`.env` 或 YAML），用户可自行扩展。

1. **分行业白名单**：为每个行业注册三个维度的信源（正面评测 / 负面风险 / 客观数据），根据 `market_segment` 自动选取
2. **W/T 专用抓取**：对每个竞品追加一条"负面/风险"搜索 query，如 `"{name} 投诉 问题 召回 亏损 {年份}"`
3. **信源可配置**：在 `.env` 或独立 yaml 中维护映射表，让用户可以按自己熟悉的领域增减站点
4. **数据充分性标记**：爬取完成后评估每个竞品的 W/T 证据密度，不足时在 SWOT prompt 中显式告知 LLM，要求它对贫瘠竞品降低密度而非编造

### 10.6 报告端与 SWOT 端的补充方向

| 维度 | 问题 | 解决方向 |
|------|------|------|
| SWOT 单次调用过载 | 5 个竞品 × 4 象限 = 20 个格子挤在 2200 token 里，每格 110 token | 拆成每个竞品独立调用，拿完整 profile 深挖；或批量调用但加硬规则——evidence 为空的条目直接丢弃不输出 |
| 报告无统稿 | executive_summary 先写，没看过后面每个 section 的内容 | 加"主编"环节：完整草稿 → 二次 LLM 调用去重、统一口径、确保术语一致 |
| 竞品对比混乱 | 三份内容（market_analyst 列表 + SWOT cross_analysis + narrative）互不引用、格式不统一 | 单独的固定格式对比矩阵 section（表格而非段落），assembly 时嵌入 |

## 11. 架构级改善：从开环流水线到闭环竞品分析（2026-06-24）

> 对照 gpt-researcher 的报告生成机制后，发现 CompIntel 的核心差距不在模型或 prompt，而在架构层面：当前是开环流水线（数据全量灌入 → 各自为政生成 → 拼接输出），缺少闭环反馈（过滤 → 验证 → 统稿）。方案是在保有竞品分析特色（意图解析、对比矩阵、SWOT、RAG 回写）的前提下，注入三个闭环节点。

### 11.1 当前架构 vs gpt-researcher 的本质差异

| 维度 | GPT-Researcher | CompIntel 当前 | 差距原因 |
|------|------|------|------|
| 大纲生成 | 检索**之前**生成大纲，搜索围绕大纲展开 | 无大纲。搜索 query 是通用模板句 | 无闭环：搜索不知道自己在为报告的哪个章节服务 |
| 内容内省 | 每爬完一页，LLM 判断相关性/可信度 → 不相关直接丢弃 | **完全没有**。爬出来的全部灌入 | 无闭环：垃圾数据一路流到终点 |
| 来源管护 | 逐源摘要 + 去重 + 权威性评估 + 多样性维护 | 无。snippet 直接贴在 profile 里 | 无闭环：LLM 被迫在 2200 token 内同时做"理解+判断+去重+写作" |
| 逐章撰写 | 按大纲章节独立写，每章只看自己相关的 sources | exec_summary → narratives → conclusion 串行，互不引用 | 无闭环：总结写在最前面，没见过后面内容 |
| 统稿 | Editor + Revisor 两个角色做内容审查和迭代改进 | 无。assembly 是纯代码拼接 | 无闭环：术语不统一、矛盾不检测 |
| 竞品对比矩阵 | **没有** — 通用研究报告 | ✅ SWOT + cross_analysis + 竞品 narrative | **CompIntel 独特优势，保留** |

### 11.2 目标架构：闭环竞品分析流水线

```
用户查询
    │
    ▼
┌──────────────────┐
│ 意图解析          │  ← 保留。竞品意图解析是独特价值
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│ 研究规划          │  ← 保留，但 prompt 改为大纲驱动
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│ fan-out 竞品画像  │  ← 保留。search + scrape + RAG 并行
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│ [新] 内省·管护     │  ← 核心新增。解决 SWOT 质量差的根源
│  curator          │
│  - 逐来源质量评分  │
│  - 去重 + 去广告   │
│  - W/T 数据充分性  │
│    评估 + 补充搜索  │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│ [新] 大纲驱动撰写  │  ← 重构 report_writer
│  - 竞品对比矩阵    │     每章独立生成，只看自己的来源
│  - 逐竞品 SWOT    │     竞品矩阵是独立章节，固定表格格式
│  - 赛道格局       │     SWOT 拆为每竞品独立调用
│  - 策略建议       │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│ [新] 主编统稿     │  ← 新增
│  - 术语统一       │     完整草稿 → LLM 去重 + 矛盾检测
│  - 去重           │     executive_summary 延迟到此处重写
│  - 矛盾检测       │
└────────┬─────────┘
         │
         ▼
┌──────────────────┐
│ 审阅              │  ← 保留，但输入从"草稿"变为"统稿后"终稿
└────────┬─────────┘
         │
         ▼
      RAG 回写
```

### 11.3 三个新增节点详解

#### 节点 1：内省·管护（curator）

**位置**：fan-out profile 收集完毕后，进入下游分析/SWOT 之前。

**解决的问题**：SWOT W/T 无 evidence；竞品对比数据混乱；垃圾数据污染下游 prompt。

**职责**：

| 操作 | 做法 | 效果 |
|------|------|------|
| 质量评分 | 轻量 LLM 调用判断每条来源：与竞品相关吗？含负面/风险信息吗？ | 广告/404/无关 → 丢弃；含批评/投诉/财报 → 标记为 W/T 高价值来源 |
| 去重 | 检测不同来源是否转载同一内容 | 保留最权威的版本 |
| W/T 充分性评估 | 每个竞品统计 S/O/W/T 各象限 evidence 数量 | 不足的象限标记 + 触发一次定向补充搜索 |
| 数据分级标记 | 每个竞品输出如 `W/T_DATA_WEAK` | 下游 agent 知道哪些竞品数据薄弱，不需要编造 |

**输出**：清洗后的高质量 profiles，附带 evidence 分布标记和补充搜索请求。

#### 节点 2：大纲驱动撰写

**位置**：替换当前 report_writer 的 `exec_summary → narratives → conclusion` 线性模式。

**当前问题**：
- executive_summary 写在最前面，没看过后面的内容
- 竞品对比信息散在 market_analyst 列表、SWOT cross_analysis、narrative 三个地方，互不引用
- 5 个竞品 × 4 象限挤在 2200 token 内

**新架构**：

```
           竞品分析大纲
          /    |    \    \
     赛道格局  对比矩阵  逐竞品SWOT  策略建议
        |       |       /|\         |
        |       |      / | \        |
        └───┬───┘     /  |  \       |
            |        /   |   \      |
         各自独立撰写（只看自己被分配到的来源）
            |        \   |   /      |
            └─────────┬─┘──┘───────┘
                      │
                   主编统稿
```

| 大纲章节 | 撰写方式 | 数据来源 | 与旧的差异 |
|------|------|------|------|
| 赛道格局 | 1 次 LLM 调用 | 清洗后的 market_analysis | 替代旧的 market_overview section |
| 竞品对比矩阵 | 1 次 LLM 调用，固定表格格式输出 | 所有竞品的 curator 后 profile | **全新**。不再由三个 agent 各自输出一段然后拼接 |
| 逐竞品 SWOT | N 次独立 LLM 调用（每个竞品 1 次） | 该竞品的完整 curator 后 profile | 旧：N 个竞品挤在 1 次调用里 |
| 策略建议 | 1 次 LLM 调用 | 全部章节的输出摘要 | 替代旧的 conclusion |

#### 节点 3：主编统稿

**位置**：所有章节撰写完成后，审阅之前。

**解决的问题**：报告内容混乱、术语不统一、矛盾不检测、executive_summary 跟不上实际内容。

**职责**：
- 检查竞品名称/术语在各章节中是否一致（如"比亚迪"在 SWOT 里叫"BYD"，在 narrative 里叫"比亚迪"）
- 检测矛盾（如"SWOT 说特斯拉品牌强势，策略建议却说特斯拉品牌力下滑"）
- 去重（同一段信息出现在两个章节里 → 保留一个版本）
- **重写 executive_summary**（此时它已能看到完整报告所有章节）
- 输出终稿

### 11.4 哪些改、哪些不改

| 组件 | 动作 | 原因 |
|------|:--:|------|
| intent_analyst | 保留 | 竞品意图解析（target + competitors + market_segment）是独特优势 |
| research_planner | 保留 | 但 prompt 改为大纲驱动，告知 LLM 报告将包含哪些章节 |
| competitor_profiler | 保留 | search + scrape + RAG 并行不变，但下游新增 curator 过滤 |
| market_analyst | 合并到 curator | 市场分析本质上就是 curator 处理完整 profile 集合后的副产品 |
| SWOT synthesizer | 拆分 | 1 次批量调用 → N 次独立调用（每个竞品 1 次） |
| report_writer | 重构 | 线性模式 → 大纲驱动 + 逐章撰写 + 主编统稿 |
| reviewer | 保留 | 输入从未统稿的草稿变为统稿后的终稿 |
| RAG 回写 | 保留 | 终稿更干净，回写质量随之提升 |

### 11.5 效果预估

| 问题 | 当前根因 | 架构修复 |
|------|------|------|
| SWOT W/T 无 evidence | 没去能产生 W/T 的地方搜 + 搜到也不标记 + 全量灌 | curator 标记 W/T 弱 → 定向补充搜索 → 下游知道数据贫瘠 |
| 竞品对比混乱 | 三份内容互不引用、格式各异 | 对比矩阵独立成章，固定表格格式 |
| 报告内容混乱 | 无统稿 + exec_summary 写在最前面 | 主编节点：术语统一 + 去重 + 矛盾检测 + 延迟重写 exec_summary |
| Prompt 膨胀 | 5 个竞品数据全塞一个 prompt | 逐章撰写，每章只拿自己需要的数据子集 |
| 垃圾数据污染 | 广告/404/无关页面全量灌入 | curator 逐来源评分 → 不相关直接丢弃 |

### 11.6 与 gpt-researcher 的差异化定位

| | GPT-Researcher | CompIntel（目标架构） |
|------|------|------|
| 定位 | 通用研究报告生成 | **竞品情报专用工具** |
| 核心输出 | 学术论文风格综述报告 | 竞品对比矩阵 + 逐竞品 SWOT + 可信度评分 |
| 特色能力 | 大纲驱动 + 内省过滤 | 大纲驱动 + 内省过滤 + **竞品矩阵** + **RAG 越用越聪明** |
| 用户交互 | Web 端 | CLI 终端（开发者原生） |

## 12. RAG 混合检索：BM25 稀疏 + Dense 语义 + RRF 融合（P4A，2026-06-25）

> 当前 RAG 仅使用纯 Dense 向量（余弦相似度），在精确术语匹配、中英文混合、短查询等场景下召回质量不足。引入 BM25 稀疏检索 + RRF 融合排序，以最小改动换取最大检索质量提升。

### 12.1 当前检索的三种失效模式

| 失效模式 | 具体场景 | 纯 Dense 为什么失败 |
|------|------|------|
| **精确名称匹配** | 查询 "BYD" vs 库中 "比亚迪" | Dense Embedding（BAAI/bge-small-zh）将 "BYD" 映射到英文子空间，"比亚迪" 映射到中文子空间，余弦相似度极低——虽然是同一个实体 |
| **稀有术语** | 查询 "黑猫投诉 特斯拉" vs 库中 "Tesla 消费者投诉" | 稀有术语的 embedding 向量在训练数据中出现极少，被池化噪声淹没。BM25 可通过精确字符串匹配直接命中 |
| **短查询** | 查询 "定价" vs 库中长文档含"freemium 定价" | 单字语义向量过于泛化，与大量无关文档共享高余弦相似度。BM25 的 TF 权重可精确匹配 |

### 12.2 为什么是 BM25 + Dense + RRF 三件套

| 组件 | 解决什么 | 为什么不能单独用 |
|------|------|------|
| **BM25（稀疏检索）** | 精确关键词匹配、稀有术语、数字/代码 | 不理解同义关系（"EV" = "电动车"），对改写后的查询失效 |
| **Dense（语义检索）** | 同义词、跨语言、语义近似 | 短查询过拟合、稀有术语被噪声淹没、精确名称匹配差 |
| **RRF（Reciprocal Rank Fusion）** | 融合两个排序列表，不要求分数同量纲 | 没有融合无法判断 "BM25#1 + Dense#10" vs "BM25#5 + Dense#3" 哪个更相关 |

RRF 公式：`score(d) = Σ 1/(k + rank_i(d))`，k=60。

### 12.3 实现策略：Qdrant 原生多向量 + RRF

利用 Qdrant 1.10+ 的命名向量 + Prefetch + Fusion 能力，改动集中在 `qdrant_store.py`：

```
Before:  collection {dense_vector}
         query_points(query=dense_vector)

After:   collection {dense: dense_vector, sparse: BM25 sparse}
         query_points(
           prefetch=[
             Prefetch(query=dense, using="dense", limit=20),
             Prefetch(query=sparse, using="sparse", limit=20),
           ],
           query=FusionQuery(fusion=Fusion.RRF),
           limit=5,
         )
```

### 12.4 稀疏方案：纯 Python BM25，零新依赖

BM25 核心公式（k1=1.5, b=0.75）：
```
score(d,q) = Σ IDF(q_i) · TF(q_i,d)·(k1+1) / (TF(q_i,d) + k1·(1-b + b·|d|/avgdl))
IDF(q_i) = log((N - df + 0.5) / (df + 0.5) + 1)
```

### 12.5 改动范围

| 文件 | 改动 | 量 |
|------|------|:--:|
| `compintel/rag/qdrant_store.py` | BM25Embedder + 命名向量集合 + hybrid_search + ingest 双写入 | ~180 行 |
| `compintel/agents/rag_retriever.py` | similarity_search → hybrid_search | ~5 行 |
| `compintel/rag/base.py` | SparseVector 类型别名 | ~5 行 |
| `tests/test_compintel_core.py` | 测试适配 | ~20 行 |

总计约 210 行变更。图管道和 agent 零感知——底层检索从 Dense 升级到 Hybrid。

## 13. 搜索节点 ReAct 化：自评估 + 自适应重搜（P1A，2026-06-25）

> 当前 SearchWorker 是一次性调用：构建查询 → 调搜索 API → 返回结果。不评估结果质量，不自适应换词重搜。搜索结果质量是整条管道的天花板——搜回来的全是 PR 稿，下游分析再强也没用。

### 13.1 当前失效模式

| 场景 | 当前行为 | 后果 |
|------|------|------|
| 搜索结果全是新闻稿/PR | 直接返回，不做判断 | 下游 market_analyst 和 SWOT 基于新闻稿做分析，产生空洞报告 |
| 搜索结果不足 3 条 | 直接返回 | 下游 profile 数据贫瘠，aggregator 只能输出 "insufficient data" |
| 搜索词太宽泛（如 "Tesla"） | 返回品牌介绍页而非竞品对比页 | 得不到任何竞争情报，profile 本质为空 |
| 搜索结果全是中文但需要英文 | 不感知语言错配 | RAG 混入无关语言的文档 |

### 13.2 为什么需要 ReAct 而非修 Prompt

修搜索 prompt（加 "competitor analysis" 后缀）只能解决 "搜索词太宽泛" 这一个问题。实际搜索质量受多因素影响：搜索引擎返回了什么内容、目标公司是否有足够的公开竞争信息、当前搜索词是否命中了正确的内容类型。这些不是 prompt 能预先知道的——需要**搜完再看**。

### 13.3 方案：两轮自评估循环

```
构建初始查询 → 调 API → 评估结果质量 →
  if 达标: 返回
  if 未达标: 换关键词重搜（最多 2 轮）→ 返回最佳结果
```

**质量评估**（纯规则，不调 LLM，零成本）：
- 结果数量 ≥ 3
- 有效摘要比例 ≥ 50%（摘要长度 ≥ 50 字符）
- 非 PR/新闻比例 ≥ 30%（标题不含 "press release"、"announces"、"新闻稿" 等标志词）

**自适应改词**（同样纯规则）：
- 追加分析关键词：`"analysis" "comparison" "vs" "行业分析" "竞争" "对比"`
- 替换角度：如 "Tesla" → "Tesla 电动汽车 市场份额 竞争对手"

### 13.4 改动范围

| 文件 | 改动 | 量 |
|------|------|:--:|
| `compintel/agents/search_worker.py` | 新增 `_evaluate_quality` + `_reformulate_query` + 重搜循环 | ~60 行 |

改动集中在一个文件内，对管道其他组件零影响。

## 14. 自动化回归评测流水线（P2B，2026-06-25）

> 当前 `evaluate.py` 可以对一份 bundle 做 7 维度打分，但没有「改 prompt → 跑 N 个历史查询 → 对比分数 → 标红退步项」的自动化流程。这直接导致"修好 A 类查询、坏了 B 类查询"的回归问题无法被发现。

### 14.1 为什么需要离线评测集

当前开发流程：改了 prompt → 手动跑一两条查询 → 看报告 → 觉得 OK → 提交。问题是：

1. **样本量 = 1 不具代表性**。修 "分析 Notion" 这条 query 的 prompt，可能让 "分析特斯拉" 那条 query 的输出变差，但你不会知道——因为没测。
2. **人工看报告效率低**。跑一次完整管道 3-5 分钟，看一份报告 2-3 分钟，10 条查询就是将近 1 小时。所以没人会全测。
3. **评分不一致**。今天觉得"这份报告 7 分"，明天可能觉得同一份报告"只能给 5 分"。人工评分有波动。

离线评测集解决这三个问题：5 分钟跑完 10 条历史查询，机器打分（7 维度 × 0-2 分），对比 baseline 自动标红退步项。

### 14.2 方案：轻量级，不依赖外部工具

```
compintel/regression_test.py
  ├── TEST_QUERIES: list[str]     ← 10 条代表性查询
  ├── BASELINE_SCORES: JSON file   ← 每条查询的 7 维分数快照
  ├── run_regression()             ← 逐条跑管道 → evaluate → diff
  └── format_diff_table()          ← 终端输出对比表格，标红退步项
```

**查询库设计**（覆盖多种场景，而非越多越好）：

| 场景 | 示例查询 | 数量 |
|------|------|:--:|
| 中文单竞品 | "分析 Notion 的竞品格局" | 2 |
| 英文查询 | "Analyze Tesla competitors in EV market" | 2 |
| 多竞品对比 | "对比 比亚迪 特斯拉 蔚来 的竞争策略" | 2 |
| 中文无竞品（依赖种子发现） | "分析 Figma" | 1 |
| 投资视角 | "分析小米在智能汽车领域的投资布局" | 1 |
| 小众市场（爬取源少） | "分析瑞幸咖啡的竞品格局" | 1 |
| 模糊查询 | "知识管理工具有哪些" | 1 |

10 条查询覆盖了系统的主要使用路径和已知脆弱点。不是越大越好——关键是代表性和覆盖度。

### 14.3 baseline.json 快照机制

```json
{
  "分析 Notion 的竞品格局": {
    "total": 10, "max": 14,
    "scores": {
      "1.竞品真实性": 2, "2.赛道准确性": 1, "3.RAG记忆利用": 1,
      "4.Source可追溯": 1, "5.降级行为": 2, "6.Pipeline完整性": 2, "7.报告可读性": 1
    }
  }
}
```

每次跑完回归测试后，如果新分数 *全部* ≥ baseline，更新 baseline。如果有退步，必须先修复再更新。

### 14.4 改动范围

| 文件 | 改动 | 量 |
|------|------|:--:|
| `compintel/regression_test.py` | 新文件：查询库 + 回归运行 + diff 输出 | ~100 行 |
| `tests/test_regression.py` | 新文件：CI 可运行的回归测试 | ~30 行 |

## 15. 节点自我审视：Market/SWOT 输出校验（P1B，2026-06-25）

> LLM 生成的 market_analysis 和 swot_analysis 可能包含不在输入 profile 中的公司名、同一公司出现在多个互斥分类中、跨章节矛盾等错误。当前这些错误一路流到最终报告，直到 Reviewer 才发现（也可能发现不了）。

### 15.1 失效案例

| 错误类型 | 实际案例 | 为什么 Reviewer 可能漏掉 |
|------|------|------|
| **幻影公司** | market_analysis 的 leaders 列出 "Notion"、"Slack"，但 Slack 根本不在输入的 5 个竞品中——LLM 从训练数据中"脑补"的 | Reviewer 不知道输入的竞品列表是什么 |
| **重复分类** | 同一公司 "Coda" 同时出现在 leaders 和 challengers 中——LLM 在不同段落做了不同判断 | Reviewer 靠 LLM 评分，LLM 自己犯了错自己再评一次——盲点 |
| **名字变异** | SWOT 中写 "BYD"，cross_analysis 中写 "比亚迪"——同一实体被当作两个不同公司 | Reviewer 不检查实体对齐 |

### 15.2 方案：纯规则自检，非阻塞，零成本

在 MarketAnalyst 和 SWOT Synthesizer 的 LLM 产出后，追加一个纯代码的自检步骤：

```
LLM 产出 → 规则自检 → 
  if 无问题: 正常返回
  if 有问题: 追加 warnings → 正常返回（非阻塞）
```

**MarketAnalyst 自检项**：
- names_in_landscape ⊆ profile_names（不凭空捏造公司）
- leaders ∩ challengers ∩ niche = ∅（不重复分类）
- 每一项 ≥ 1 个公司

**SWOT 自检项**：
- swot_names ⊆ profile_names
- cross_analysis 中引用的公司名全部在 swot_names 中

全部纯集合运算，不调 LLM，耗时 < 1ms。

### 15.3 为什么非阻塞

自检发现的问题不能自动修复——比如 LLM 捏造了一个公司名 "Figma"（它不在输入竞品列表中，但它确实是真实公司，可能是 LLM 基于常识推断的）。这种情况下，警告比丢弃更合理：下游 ReportWriter 看到 warning 后可以在叙述中标注 "需人工验证"。

### 15.4 改动范围

| 文件 | 改动 | 量 |
|------|------|:--:|
| `compintel/agents/market_analyst.py` | `__call__` 注入自检 + `_self_audit` 方法 | ~30 行 |
| `compintel/agents/swot_synthesizer.py` | `__call__` 注入自检 + `_self_audit` 方法 | ~30 行 |

## 16. Prompt 注册表：模板外置 + 版本化（P2A，2026-06-25）

> 当前所有 agent 的 prompt 以 f-string 硬编码在 `__call__` 方法中。改一个字 = 改代码 = 无 diff review = 无 A/B test = 无回滚。提取到 YAML 文件中，实现版本可追溯、变更可审计。

### 16.1 设计原则：轻量实用，不引入框架

| 决策 | 理由 |
|------|------|
| YAML 格式 | 比 JSON 好读（多行字符串、注释），比 .py 文件好 diff |
| 每个 agent 一个 YAML | 独立版本号，改一个不影响另一个 |
| `model_key` + `max_tokens` + `temperature` 写入 YAML | Prompt 和参数一同版本化——改 prompt 时经常也需要调参数 |
| 变量插值用 `{query}` 占位符 | 与 Python f-string 语法一致，调用方零学习成本 |
| 不引入 Jinja2 | 不需要条件/循环。简单的 `str.format()` 足够 |
| 可选加载：未找到文件时回退到内置默认 | 零破坏性——没有 YAML 文件也能正常工作 |

### 16.2 YAML 文件结构

```yaml
# compintel/prompts/market_analyst.yaml
version: "1.0.0"
model_key: smart
max_tokens: 1800
temperature: 0.2
system: null
prompt: |
  You are CompIntel's market analyst.
  Analyze the SPECIFIC market segment named below and return strict JSON...
  Market segment: {market_segment}
  Profiles: {profiles}
```

| 文件 | 改动 | 量 |

## 18. 推理模型 formatter 失败：原因 + 修复方案（2026-06-25）

> 线上运行时报错：`Formatter call failed: LLM reasoning model returned only chain-of-thought without a JSON object. The model consumed all tokens on reasoning.` 错误发生在 `call_with_reasoning` 的 Phase 2（格式化），而非 Phase 1（推理）。

### 18.1 错误链路

```
Phase 1: deepseek-reasoner
  → create_chat_completion_raw() 返回 {reasoning_content: 4000 chars, content: ""}
  → reasoning_text 正常捕获 ✓

Compression bridge:
  → _compress_reasoning() 调 deepseek-chat 压缩推理文本 → 成功 ✓

Phase 2: deepseek-chat (formatter)
  → create_chat_completion() → _post_chat_completion_raw()
  → 返回 {reasoning_content: "...", content: ""}  ← 非推理模型也出了推理字段!
  → llm.py:167-178: content 空 + reasoning 非空 → _extract_tail_json(reasoning)
  → 推理文本中无 JSON 对象 → RuntimeError 抛出
  → call_with_reasoning Phase 2 重试 3 次，全部失败
  → call_with_reasoning 返回 None
  → 下游 agent 退至 derived/template（完全放弃 LLM 质量）
```

### 18.2 根因分析

**表面原因**：`deepseek-chat`（非推理模型）在某些 prompt 下也产出了 `reasoning_content` 并将 `content` 留空。这可能是因为 DeepSeek 服务端行为变更（部分 chat 模型开始采用先思考后回答的模式），也可能是格式化 prompt 过长导致模型进入推理状态。

**深层原因**：`call_with_reasoning` 的两阶段设计缺少内部降级——formatter 失败时直接返回 `None`，导致整条推理 pipeline 的成果被丢弃。

### 18.3 修复：formatter 失败时回退到直接 call_and_parse

在 `call_with_reasoning` Phase 2 的 retry 循环全部耗尽后，不回退到 `None`，而是尝试用 `call_and_parse` 直接调用格式化模型（传原始 prompt，不用推理上下文）。这相当于：

```
Phase 1 (reasoner) → 成功：推理文本可用
Phase 2 (formatter+reasoning) → 失败
  ↓ 回退
Phase 2' (formatter only) → call_and_parse(original_prompt)
  → 成功：返回解析后 JSON（无推理增强，但仍为 LLM 质量）
  → 失败：返回 None（此时才走 derived/template 降级）
```

这是两阶段管道内部的优雅降级——推理产出仍然被记录在日志中，格式化失败不丢弃 LLM 质量。

### 18.4 改动范围

| 文件 | 改动 | 量 |
|------|------|:--:|
| `compintel/core/llm_service.py` | `call_with_reasoning` Phase 2 retry 耗尽后追加 `call_and_parse` fallback | ~15 行 |

## 19. 模型名迁移：deepseek-chat/reasoner → deepseek-v4-pro/flash（2026-06-25）

> 用户用量面板中只看到 `deepseek-v4-pro` 和 `deepseek-v4-flash` 的 token 消耗，没有 `deepseek-chat` 或 `deepseek-reasoner`。查证 DeepSeek 官方文档后确认：`deepseek-chat` 和 `deepseek-reasoner` 是**旧版别名**，将于 2026-07-24 停用。

### 19.1 模型映射关系

| 代码当前使用的旧名 | DeepSeek 后端实际路由 | 2026-07-24 后 |
|---|---|---|
| `deepseek-chat` | `deepseek-v4-flash`（非思考模式） | **失效** |
| `deepseek-reasoner` | `deepseek-v4-flash`（思考模式） | **失效** |

V4 时代不再用不同模型名区分思考模式，而是统一模型名 + `thinking_mode` 参数（`non-thinking` / `thinking` / `thinking_max`）。

### 19.2 这也解释了 formatter 失败

`deepseek-reasoner` = `v4-flash` + 思考模式。当 Phase 2 的 `deepseek-chat`（也是 `v4-flash`）被传入较长的推理上下文后，某些情况下模型也会进入思考模式——把所有 token 烧在 chain-of-thought 上，`content` 留空。

V4 API 的正确做法：
- 需要推理 → 调 `deepseek-v4-pro` + `thinking_mode: "thinking"`
- 需要格式化 → 调 `deepseek-v4-flash`（默认 non-thinking）

两个阶段用**不同模型**而非依赖别名隐式切换。

### 19.3 迁移方案

| 配置项 | 旧值 | 新值 | 说明 |
|------|------|------|------|
| `FAST_LLM` | `deepseek-chat` | `deepseek-v4-flash` | 快速任务（意图解析、规划） |
| `SMART_LLM` | `deepseek-chat` | `deepseek-v4-pro` | 智能任务（市场分析、编辑） |
| `STRATEGIC_LLM` | `deepseek-chat` | `deepseek-v4-pro` | 战略任务（SWOT、审阅） |
| `REASONING_LLM` | `deepseek-reasoner` | `deepseek-v4-pro` | 推理阶段（+ `thinking_mode`） |
| `FORMATTING_LLM` | `deepseek-chat` | `deepseek-v4-flash` | 格式化（非思考模式） |

`call_with_reasoning` 的两阶段调用改为：
- Phase 1：`deepseek-v4-pro` + `thinking_mode: "thinking"`
- Phase 2：`deepseek-v4-flash`（默认 non-thinking）→ 不会再无故进入思考模式

### 19.4 改动范围

| 文件 | 改动 | 量 |
|------|------|:--:|
| `compintel/settings.py` | 5 个模型默认值更新 | ~5 行 |
| `.env.example` | 模型名 + 新增 `THINKING_MODE` 说明 | ~10 行 |
| `compintel/llm.py` | 新增 `thinking_mode` 参数透传 | ~10 行 |
| `compintel/core/llm_service.py` | Phase 1 传 `thinking_mode="thinking"` | ~5 行 |

## 20. Tavily SSL 握手失败 + Formatter CoT 燃尽：两个反复出现的运行时故障（2026-06-25）

> 真实运行中两个问题反复出现，已在优化文档中多次记录相关修复但未根除。本节聚焦根因与最终修复。

### 20.1 Tavily SSL：`SSLEOFError` — Windows 下的 SSL 证书问题

**现象**：
```
HTTPSConnectionPool(host='api.tavily.com', port=443): Max retries exceeded
Caused by SSLError(SSLEOFError(8, '[SSL: UNEXPECTED_EOF_WHILE_READING]'))
```

`curl -I https://api.tavily.com` 正常返回 200，排除网络/防火墙问题。

**根因**：`tavily-python` SDK 内部使用 `httpx` 或 `requests` 发起 HTTPS 请求。Windows 上 Python 的 SSL 证书验证链与系统证书存储之间存在已知兼容性问题——尤其是在 Conda 环境或企业网络下。`curl` 使用自己的证书包不受影响，但 Python 的 `ssl` 模块可能找不到正确的 CA bundle。

更关键的是，`search_worker.py` 的 ReAct 循环每轮发起请求（最多 3 轮），每轮都新建 `TavilyClient` 实例——SSL 握手失败的概率在多次重试中累积放大。

**修复方向**：
1. 给 `TavilyClient` 传入 `verify` 参数或自定义 SSL context
2. 或降低 ReAct 对 Tavily 的依赖——搜索失败时不重试同 provider，直接走降级路径

### 20.2 Formatter CoT 燃尽：V4 模型的 reasoning 行为不可预测

**现象**：
```
Formatter call failed (1/3): LLM reasoning model returned only chain-of-thought
without a JSON object. The model consumed all tokens on reasoning.
```
→ 3 次重试全失败
→ fallback `call_and_parse` 也失败（同原因）

**根因分析**：

DeepSeek V4 系列模型的 `thinking_mode` 行为不完全可控。即使：
- Phase 1 用 `deepseek-v4-pro` + `thinking_mode="thinking"`（应该有推理产出）
- Phase 2 用 `deepseek-v4-flash`（默认 non-thinking）

实际上 V4 模型在遇到**长 prompt**（压缩后的推理上下文 + 格式化指令）时，可能自发进入推理状态——把所有 token 烧在 CoT 上，`content` 留空。

此外，用户本地 `.env` 若仍使用旧模型名（`deepseek-chat` / `deepseek-reasoner`），旧别名映射到 v4-flash 的不同模式，行为与预期一致。

**修复方向**：
在 `llm.py` 的 `create_chat_completion` 中，当 content 为空但 reasoning 非空时，尝试从 reasoning 尾部提取 JSON（如同 `create_chat_completion_raw` 已做的），而非直接抛 RuntimeError。这样：
- Phase 1（reasoner）：推理文本被捕获 → 压缩后传入 Phase 2
- Phase 2（formatter）：若 content 为空，从 reasoning 提取 JSON → 若提取成功则正常返回 → 若提取失败则优雅降级

### 20.3 改动范围

| 文件 | 改动 | 量 |
|------|------|:--:|
| `compintel/agents/search_worker.py` | TavilyClient 传入 `verify=True` + 错误处理增强 | ~10 行 |
| `compintel/llm.py` | `create_chat_completion` 空 content 时回退到 reasoning 尾部 JSON 提取 | ~15 行 |
